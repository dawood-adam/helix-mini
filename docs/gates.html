Gates & control

How a run pauses, asks, sends itself back, restarts itself, and resumes.

What is a gate

A gate is the decision point at the end of every stage. loop.advance:

  1. Runs the stage (the agent answers; the snapshot is minted; the report is written).
  2. Calls gates.decide_gate(state, stage, decision, rationale, ask=, interactive=, plan=).
  3. Feeds the result into transitions.next_stage — the one routing authority.

decide_gate consults the run-scoped Plan to decide whether to ask the human (HITL) or auto-proceed. An ask goes through the client-IO seam as MCP elicitation; the choice the human makes is translated into a typed GateDecision.

The three decisions

Whether the gate is auto or HITL, it always resolves to one of:

DecisionGateDecisionEffect
proceed("proceed", None, None)Advance to the next stage in pipeline order. END past the last stage.
send back("goto", target, note)Jump to target (any stage), recording note as feedback for that stage's next run.
stop("stop", None, None)End the run cleanly at END. Resumable from the last snapshot.

HITL prompt shape (MCP elicitation, scripted by io.gate_asker):

After scout_critic: 2 candidate approaches; recommends approach-1.
Strengths: rPPG well-studied; weaknesses: rejection method.
Note: this run is auto-proceeding until 'builder'.

[ proceed · send back · stop ]
   │
   └── send back → which stage? [ scout · scout_critic · planner · … ]
                    feedback for that stage (one line)?

Declining the gate elicitation raises Declined, which the orchestrator catches and turns into a resumable paused-input pause — same contract as the cost-ceiling pause.

The Plan

Run control is the run-scoped Plan (core.plan.Plan). It is threaded through the loop alongside the gate asker — never stored in PipelineState, so it never lands in a snapshot. The Plan governs two things at each transition:

  • Autonomy per gate: auto or ask.
  • Directive per stage: an optional one-line instruction injected into that stage's USER prompt on the next run (through the existing record_feedback channel).

Two construction paths:

ConstructorResult
Plan()Empty — ask at every gate (step-by-step).
Plan.from_autonomy_until(s)Compat. "" / "none" → pure HITL; a stage name → auto until that stage; "END" → full auto; "rediscover" → full auto + the discovery loop (see below).
Plan(steps=[Step("scout", autonomy="auto"), …])Per-stage autonomy + optional directive per stage.

Autonomy modes

ModeBehaviourWhen to use
Step-by-stepAsk at every gate. Default for the wizard.First run on a new project; exploratory work; teaching.
autonomy_until="builder"Auto-proceed Scout / Scout Critic / Planner gates; pause at the Builder.You trust the framing but want to review the code before metrics.
autonomy_until="END"Auto-proceed every gate; deterministic routing only.Reproducing a known result; batch over many projects.
autonomy_until="rediscover"Like END, plus the hypothesis loop — see below.Open-ended exploration with bounded spend.
Explicit stepsA [Step(agent, autonomy, directive?)] list. Stages not in the plan fall back to ask.Programmatic control; targeted directives per stage.

Send-back, in detail

Send-back is the only correction mechanism — there is no separate "edit" or "rewind". When you pick send back at a gate:

sequenceDiagram participant U as You (via Claude Code) participant G as gates.decide_gate participant S as state.human_feedback participant N as transitions.next_stage participant L as loop.advance G->>U: elicit — proceed / send back / stop? U-->>G: send_back G->>U: elicit — which stage? U-->>G: scout G->>U: elicit — feedback for that stage? U-->>G: restrict to 2024 papers G->>S: record_feedback(from=scout_critic, to=scout, note) G-->>N: GateDecision("goto", "scout", note) N-->>L: next_stage = "scout" L->>L: mint a snapshot for the send-back itself L->>L: re-run scout — its _ctx_scout includes the new feedback block

Two things happen at once. The directive is appended to state.human_feedback targeting the chosen stage; the send-back itself is captured as a snapshot, so the branch you took is fully resumable.

The receiving stage's context builder (core.agents._ctx_<stage>) ends with a _feedback_block(s, stage) call that renders the directive into the USER prompt as: 

## Human feedback — address this on this pass
- restrict to 2024 papers

The agent reads the block on its re-run. There is no special "send-back agent" or alternate code path; every send-back goes through the same next_stage + the same advance as a normal transition.

Deterministic routes

Two routes fire regardless of the Plan's autonomy:

TriggerEffect
Validator emits at least one HARD: flagRoute back to Builder; the flags appear in Builder's _ctx_builder as Validator flags.
Results Critic returns verdict: "iterate"Route back to Builder; the critic's recommendations appear as feedback.

These are deliberate. A run that auto-proceeded to the Validator and failed shouldn't be forced through a manual gate just to be told to iterate; the orchestrator routes it back with the flags already threaded in.

The rediscover loop

autonomy_until="rediscover" activates the autonomous hypothesis loop (Workstream H). After Results Critic, if the iteration count is below the cap (REDISCOVER_CAP = 3), the orchestrator overrides the END transition to scout:

flowchart LR S[Scout] --> SC[Scout Critic] --> P[Planner] --> B[Builder] --> V[Validator] --> RC[Results Critic] RC -- "iter < 3" --> S RC -- "iter == 3 or budget hit" --> END([END]) classDef pipe fill:#ddf4ff,stroke:#0550ae; class S,SC,P,B,V,RC pipe

The carrier across iterations is the hypothesis thread:

  • Scout proposes hypotheses on the first iteration; subsequent iterations refine the survivors and add new candidates.
  • Scout Critic ranks them on testable / falsifiable / distinct (the idea tournament — Co-Scientist shape).
  • Results Critic returns a per-hypothesis verdict (supported · refuted · inconclusive) and writes revised statements back to the thread.

Three guard rails keep spend bounded:

  1. Hard iteration cap: REDISCOVER_CAP = 3 in core.hypothesis.
  2. Token / call budget: the existing [limits] apply across iterations; reaching either pauses the run with a resumable snapshot.
  3. Hypothesis status machine: refuted / superseded hypotheses are pushed to the bottom of the ranking, so a misranked idea can't dominate forever.

Lint check discovery-log flags any hypothesis marked Supported with no support: citations — a discovery without evidence.

Steering mid-run

The Plan is live for the duration of a run, held in runs._LIVE. hx_run_plan_set mutates it; the new plan is effective at the next gate (the loop re-reads the Plan at every transition). 

# Change autonomy_until mid-run:
hx_run_plan_set(project="bp-from-rppg", autonomy_until="END")

# Or supply explicit steps:
hx_run_plan_set(project="bp-from-rppg", steps_json=json.dumps([
  {"agent": "scout",   "autonomy": "auto"},
  {"agent": "planner", "autonomy": "hitl", "directive": "use 5-fold CV"},
  {"agent": "builder", "autonomy": "auto"},
]))

Use this to escalate ("autonomous until I say otherwise") or de-escalate ("ask me at the next gate") without restarting the run.

Pauses & resume

Every stop has a snapshot. Five things can pause a run:

CauseSnapshot decisionResume with
Token / call ceiling reachedpausedhx_step (re-asks to raise ceiling, interactively).
Gate elicitation declined(prior stage snapshot is the resume point)hx_step resumes at the same gate.
MCP client dropped mid-stageerrorresume_pipeline(project, snap, at=).
Stage errorerrorSame as above — fix the cause, then resume_pipeline.
Server restart between hx_step and hx_submit(pending step persisted to .helix/runs/<project>.pending.json)Next hx_step(folder) re-shows the outstanding step.

hx_step(folder) is idempotent: a pending step is re-shown verbatim with the same pending_token. There is no second runner — the same advance picks up from the prior snapshot. See Snapshots for how resume reconstructs state, and Usage → Observing & resuming for the day-to-day commands.
