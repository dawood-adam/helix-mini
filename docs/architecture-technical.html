Technical architecture

Module boundaries, request flow, and the invariants that keep the codebase small.

The three layers

Helix is built as three concentric layers. The outer two are thin; the inner one is the system.

LayerModuleImportsResponsibility
Drive surface helix/mcp/ mcp SDK, helix.core, helix.io One stdio MCP server (FastMCP). Tools / resources / prompts. The only module that imports the SDK.
Orchestrator helix/orchestrator/loop.py helix.core, helix.io, helix.llm One runner. advance is the unit: run a stage → snapshot → consult the gate → transition. step_begin / submit_stage / resume_step all re-enter it.
Core helix/core/ stdlib, yaml, (optionally fastembed for embed.py) Pure pipeline + Atlas + snapshots + threads + spec + hypothesis + lint. Dependency-light: imports neither mcp nor fastembed at module level.

Invariants

These are enforced by tests and by the module structure itself:

  1. Dependency-light core. helix.core, helix.io, and helix.orchestrator.loop import without the mcp SDK or fastembed. SDK contact is confined to helix/mcp/; the embedding model is lazy-imported inside helix.core.embed.
  2. One router. All pipeline routing goes through core.transitions.next_stage. Add routing logic there, never in the orchestrator.
  3. One runner. loop.advance is the unit. step_begin / submit_stage / resume_step all re-enter it; there is no parallel pipeline.
  4. A snapshot never calls a model. mint_snapshot serializes state and content-addresses artifacts; it stores the stage's Decision Card as the human digest. Reports are rendered from the card + threads + spec (also model-free).
  5. Agent-driven, no credentials. The model is the client agent, reached only through the tool loop (hx_step / hx_submit) via the JSON-responder seam (llm.use_responder, a contextvar). No server-side sampling; nothing holds API keys.
  6. Run control is run-scoped. The Plan (core.plan) is threaded through the loop alongside ask / interactive. It is never a PipelineState field, so it never lands in a snapshot. autonomy_until survives as a compat constructor.
  7. Sandbox-first. Model-controlled strings reaching the filesystem are confined by sandbox: page / artifact content via sanitize_atlas_writes / sanitize_code_artifacts; project / run / thread / bundle / spec names via validate_project_name at every filesystem path root.
  8. One page scan. core.atlas.iter_pages is the canonical walk. atlas_index, embed, recall, and lint all adapt it rather than re-walking the tree.

The client-IO seam

helix/io.py is the single place anything is sent back to the client. The seam has one callback today: 

class ClientIO(Protocol):
    def elicit(self, req: ElicitRequest) -> ElicitResult: ...

Schemas are flat (string / enum / boolean / array), built by ask_text / ask_choice / ask_multi / ask_confirm. A run binds an implementation for its duration via use(io); any component reaches it via current(). The pipeline core is synchronous and deep; the MCP session is asynchronous. The seam bridges them through an anyio worker thread.

helix/mcp/client_io.py is the only real implementation. Tests inject a scripted IO (a queue of ElicitResult) or drive the step loop directly with a pre-baked answer.

The model is not on this seam. It is the client agent, reached through the tool loop via the JSON-responder seam in helix/llm.py (a contextvar set by the loop's _render_responder and _inject_responder).

A step: loop.advance

loop.advance(state, ctx, atlas, current, last_id, *, ask, plan, branch) is the unit. It does four things in order, exactly once per call:

  1. Run the stage. stages.run_stage → agents.run_agent. The stage's model call routes through the bound JSON responder. In agent-driven mode the responder raises NeedsModel the first time, which unwinds out of advance before any Atlas write or snapshot — the prior snapshot is the resume point. Deterministic stages (the Validator) run straight through with no round-trip.
  2. Mint a snapshot. mint_snapshot serializes state, content-addresses artifact bytes, and stores the stage's Decision Card. Zero model calls.
  3. Write the HTML report. _write_report renders a self-contained per-stage HTML file at atlas/projects/<id>/reports/<stage>.html (overwritten each run; history lives in the snapshot DAG). Best-effort: a renderer failure logs and skips, never breaks the run.
  4. Resolve the gate + transition. gates.decide_gate consults the Plan; an ask routes through the client-IO seam as elicitation. The decision feeds transitions.next_stage, which returns the next stage, an earlier stage (send-back), or END.

Agent-driven (no sampling)

There is exactly one model-consumption site in the whole server: core.agents.run_agent → llm.call_llm_json. Inverting that one seam makes the entire server run through the client agent. The other subsystems — Atlas, snapshots, recall, ingest, lint, gates, transitions, runs, hot — were already deterministic.

Two responders, one contextvar:

def _render_responder(stage):
    def _r(model, system, user):
        raise NeedsModel(stage, system, user)
    return _r

def _inject_responder(answer):
    def _r(model, system, user):
        parsed = parse_json_text(answer) if isinstance(answer, str) else answer
        tokens = (len(system) + len(user) + len(json.dumps(parsed))) // 4
        return parsed, int(tokens)
    return _r

hx_step binds the render responder and pins the resulting prompt to the pending step. hx_submit rehydrates state from the prior snapshot and binds the inject responder, so when run_agent re-runs the same stage, the model call returns the agent's submitted JSON where the model answer would have been. The atlas write, snapshot, gate, and transition all proceed unchanged.

Run control + registry

helix.runs is a small persistence layer that lives outside the pipeline core (it imports helix.config and helix.sandbox.validate_project_name, never helix.core):

  • Per run, on disk: .helix/runs/<run_id>/record.json (status, current stage, last snapshot, tokens) and events.jsonl (transition log). Survives a server restart.
  • Per project, on disk: .helix/runs/<project>.pending.json — the suspended step (run_id, stage, resume_from, branch, pinned system, pinned user, token, autonomy, clarifications). Picked up by the next hx_step.
  • In-memory: a _Live holds the run's live Plan so hx_run_plan_set can steer it and hx_run_status / hx_run_events can observe it.

Live continuation is via snapshots and resume_pipeline, not the registry. The registry adds observability and survives a restart; it is not a runtime.

Sandbox model

The threat model is straightforward: treat every model-generated string as untrusted, including ones the user-friendly elicitation produces (the model is the agent; prompt-injection in ingested sources can influence both).

SurfaceValidatorWhat it guards
Atlas page writessandbox.sanitize_atlas_writesPath under {sources,concepts,entities,projects}/, no traversal, length caps; title sanitised; content size-capped + control-stripped; action normalised; provenance whitelisted; spec_refs / aliases / links cleaned.
Builder code artifactssandbox.sanitize_code_artifactsValidated relative names under the artifacts dir; size-capped content; never executed.
Project / run / bundle namessandbox.validate_project_nameNo path separators, no .., no leading dot/hyphen; length cap; control chars stripped. Used by runs.py, snapshots.py, hot.py, core/spec.spec_path, core/threads._resolve_path.

config.workspace_root() resolves the workspace marker (helix.toml [workspace] via ancestor walk-up, or HELIX_ATLAS, or HELIX_HOME, or the run root as a final fallback). config.helix_dir() is per-project (operational state stays close to the project). config.use_root(path) is a contextvar that scopes resolution for a single run.

Test architecture

Two harnesses, three layers:

LayerHarnessCovers
Core (pure)Direct pytestspec.py, threads.py, reports.py, constitution.py, hypothesis.py, atlas action semantics, the 11 lint checks, the TDD task guardrails.
Pipelineapp.run + conftest.fake_llmEnd-to-end stage walk with a fake LLM (routed by system-prompt keyword). Verifies the agent-driven inversion, the rediscover loop iteration cap, the Plan-driven autonomy modes.
MCP drive surfacemcp.shared.memory.create_connected_server_and_client_session + scripted elicitation_callbackTools end-to-end: hx_step → feed JSON → hx_submit, looping to the done summary. Resources read back (thread://, hot://, etc.).

Optional-dependency paths (mcp, fastembed) are guarded with pytest.importorskip; the surrounding logic is tested model-free via injected functions.

Run the suite: 

pip install -e '.[dev]'
pytest -q                       # 206 passing, 1 skipped (optional fastembed)

# In a git worktree the editable install resolves to the main checkout:
PYTHONPATH="$PWD/src" pytest -q
