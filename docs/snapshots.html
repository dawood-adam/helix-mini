Snapshots — version control

A content-addressed DAG of every transition. Branchable, freezable, forkable, resumable.

What is a snapshot

A snapshot is a saved point in a run: the full PipelineState at one transition, stamped with the stage, timestamp, parent, and branch. A run opens with a start snapshot (so it is resumable before the first stage even completes), then Helix mints one after every stage and every send-back. The history is a real DAG — inspectable, diffable, branchable, and resumable from any point.

gitGraph commit id: "start" commit id: "scout" commit id: "scout_critic" branch rppg-v2 commit id: "scout (send-back, restricted)" commit id: "scout_critic" checkout main commit id: "planner" commit id: "builder" commit id: "validator" commit id: "critic_results — ship"

In this trace the gate after the first Scout Critic was a send-back to Scout with a directive ("restrict to 2024 papers"); the orchestrator branched at rppg-v2, re-ran Scout and the Critic on that branch, and the user later checked out the main line. Both branches remain resumable.

Zero cost

A snapshot calls no model. It serializes state and stores the stage's Decision Card as the human-readable digest — content the stage already produced. Artifact bytes are content-addressed under .helix/snapshots/<project>/objects/<sha256> and referenced by hash; identical files across iterations are stored once, and the state record never inlines bytes.

A snapshot stays a few kilobytes after hundreds of cycles, which is why per-stage snapshots are affordable and cycling has no fixed cap beyond the budget.

Storage

.helix/snapshots/<project>/
├── <id>.json        one snapshot (state + Decision Card + manifest)
├── index.json       the ordered DAG (id, parent, branch, stage, ts)
├── refs.json        {tags: {…}, branches: {…}}
└── objects/<sha>    content-addressed artifact bytes

A snapshot file holds: 

{
  "id": "7",
  "ts": "2026-05-19T15:42:00+00:00",
  "stage": "critic_results",
  "parent": "6",
  "branch": "main",
  "report": {"decision": "verdict: ship", "rationale": "…"},
  "decision_card": { … the Decision Card the stage produced … },
  "state": { … serialized PipelineState, with artifact bytes replaced … },
  "manifest": {
    "src/sim.py":       "8a3d…",     // sha256 references into objects/
    "tests/test_x.py":  "f12b…"
  }
}

index.json is the DAG: ordered tuples of (id, parent, branch, stage, ts). refs.json holds named tags (via hx_snap_freeze) and named branches (via hx_snap_branch).

Inspect

ToolShows
snapshot_list(project)The DAG, git-log style.
snapshot_show(project, id)One snapshot's key state.
snapshot_diff(project, a, b)Tracked differences between two snapshots.
snapshot_timeline(project)A Mermaid gitGraph of the history (paste into a markdown viewer to render).

You can also read a snapshot's metadata directly via MCP resource:

snapshot://<project>/<snap_id>       # returns the JSON metadata

Branch, freeze, fork

ToolEffect
hx_snap_branch(project, snapshot, name)Name a branch ref at a snapshot. Continuation happens by resuming with that branch name; the run lands on a new branch parented at that snapshot, keeping experiments off the main line.
hx_snap_freeze(project, snapshot, tag)Tag a snapshot immutable for publication. The reference lives in refs.json/tags.
hx_snap_fork(project, name)Export the full snapshot history (snaps + objects + index + refs) as forks/<name>.tar.gz — one self-contained, reproducible bundle.

A branch ref names a point; the branch comes to life when you resume with it.

Resume & revert

Two complementary operations:

ToolWhat it does
resume_pipeline(project, snapshot, at?, branch?, folder?) Rebuild the snapshot's full state (artifact bytes from the object store), re-enter the agent-driven loop at the snapshot's stage by default (or at= any other stage), on branch= a new branch if you want to fork the experiment. folder= roots it at the project regardless of the server's cwd. Returns the next stage's prompt (or the done summary).
snapshot_revert(project, snapshot) The file-level checkout: write a snapshot's artifacts back to disk and run nothing. Use to reproduce the on-disk state of an old run without re-entering the loop.
flowchart LR L[list_snapshots] --> S[snapshot_show id=5] S --> D{decide} D --inspect--> X[diff snap-5 snap-7] D --keep going--> R[resume_pipeline branch='rppg-v2'] D --reproduce--> V[snapshot_revert] D --share--> F[hx_snap_freeze 'v1' & hx_snap_fork 'v1-public']

There is no separate checkout: resume_pipeline covers branched continuation, snapshot_revert covers tree restoration. Together they cover both cases without a redundant destructive operation.

Pauses

A run that hits the token/call ceiling, has its gate prompt declined, or loses the MCP client mid-run records a resumable snapshot and stops rather than failing. hx_run_status reports where it stopped; resume_pipeline, or simply calling hx_step again, continues it. See Gates → Pauses for the full list of pause causes and how they map to snapshots.

The fork bundle

hx_snap_fork(project, name) packages .helix/snapshots/<project>/ in full: 

forks/<name>.tar.gz
└── snapshots/<project>/
    ├── *.json
    ├── index.json
    ├── refs.json
    └── objects/<sha>

A collaborator extracts the bundle into .helix/, points an MCP client at the project folder, and runs hx_run_status / snapshot_list / resume_pipeline to reproduce or continue the work — no shared Atlas needed (every page citation references the workspace-local Atlas, which the recipient may or may not have, but the snapshot trail and artifacts are self-contained).

For a publication-ready handoff:

  1. hx_atlas_lint and resolve blockers.
  2. hx_atlas_promote(ids, "published") — confirms via elicitation.
  3. hx_snap_freeze(project, snapshot, tag) — mark the snapshot immutable so the bundle's content addresses won't drift.
  4. hx_snap_fork(project, name) — exports the bundle. Share that one file.

Or invoke helix_freeze — the canonical MCP prompt — to walk the same checklist interactively.
