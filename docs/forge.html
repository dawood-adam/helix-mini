Forge — the pipeline

Six stages, with a human checkpoint at every transition. Agent-driven; the server runs no model.

Overview

flowchart LR S([Scout]) --> SC([Scout Critic]) SC --> P([Planner]) P --> B([Builder]) B --> V([Validator]) V --> RC([Results Critic]) V -.->|hard miss| B RC -.->|iterate| B RC -.->|rediscover| S classDef llm fill:#ddf4ff,stroke:#0550ae,color:#0550ae; classDef det fill:#dafbe1,stroke:#1a7f37,color:#1a7f37; class S,SC,P,B,RC llm class V det

Five LLM stages (blue) and one deterministic stage (green). Solid edges are the normal forward path; dashed edges are deterministic routes the orchestrator takes without asking:

  • hard miss — the Validator found at least one HARD: … band miss; route back to Builder with the flags as feedback.
  • iterate — Results Critic returned verdict: iterate; route back to Builder with the recommendations.
  • rediscover — the run is in autonomy_until: "rediscover" mode; after Results Critic, restart from Scout (capped at 3 iterations).

Stages are markdown files in src/helix/builtin_agents/. A project may override any of them with <project>/agents/<stage>.md — no code change. Frontmatter declares the stage order, kind, whether the stage may write the Atlas, and when the snapshot is minted; the body is the system prompt + output contract.

The Decision Card

Every stage emits one structured object alongside its domain output: 

{
  "summary":             "2-3 sentences: what you did + the key decision",
  "key_findings":        ["…", "…"],
  "assumptions":         ["…"],
  "open_questions":      ["…"],
  "directive_for_next":  "what the next stage should focus on",
  "confidence":          "low | medium | high"
}

The Decision Card is the single human-readable record of a stage. It is:

  • Stored verbatim in the snapshot (so snapshotting stays zero-LLM).
  • Read aloud at the gate.
  • The basis for the per-stage HTML report.
  • Used to rebuild the project hot cache at run end.

If a model omits or mangles the card, DecisionCard.from_response derives a generic one so a run never blocks on a missing field.

The six stages

1. Scout model

Scout reads the Atlas and the source folder, frames the question, and produces the project spec. It walks five sub-phases in one turn, using hx_ask for any named, recordable question:

  1. Skip option. First, asks: "Skip the interview and run full Scout end-to-end?" If accepted, jumps straight to Synthesize + Spec.
  2. Frame. Fills in FINER (Feasible · Interesting · Novel · Ethical · Relevant), PICOT (Population · Intervention · Comparator · Outcome · Time), and GQM (Goal · Questions · Metrics) in the spec frontmatter. Unanswered axes become [NEEDS CLARIFICATION: …] markers in the body — hx_clarify walks them next.
  3. Source. Cross-references the new source files against the existing Atlas (hx_atlas_recall). Each source becomes a sources/<slug>.md write with the full Workstream-G write protocol (action / because / provenance / spec_refs).
  4. Lit dive. Uses WebSearch / WebFetch (and a structured-extraction skill like superpowers if installed) to fill in gaps the sources leave. Seeds the glossary thread with canonical terms.
  5. Synthesize → Spec. Produces 2–3 candidate approaches. Writes the spec at projects/<id>/spec.md with FINER / PICOT / GQM filled in. Sets gate.status: ready only when no clarification markers remain. Before submit, runs hx_question_check — it must report ready.

Output contract (the JSON the agent submits):

{
  "source_summaries": [{"file": "…", "summary": "…"}],
  "approaches": [
    {"id": "approach-1", "title": "…", "description": "…",
     "feasibility": "high|medium|low"}
  ],
  "atlas_writes": [
    {"path": "sources/<slug>.md", …, "action": "ADD", "because": "…",
     "provenance": {"stage": "scout", "run_id": "…", "sources": ["…"]},
     "spec_refs": ["spec:0001:B"]},
    {"path": "projects/<id>/spec.md", "title": "Spec", "content": "…",
     "action": "ADD", "because": "Scout final sub-phase",
     "provenance": {"stage": "scout"}, "spec_refs": ["spec:0001:B"]}
  ],
  "decision_card": { … }
}
Spec gate
The Scout context builder calls question_check on every pass and surfaces blocker findings inline in the USER prompt, so the agent can iteratively fill in FINER / PICOT / GQM and resolve [NEEDS CLARIFICATION] markers across send-backs.

2. Scout Critic model

The Scout Critic red-teams Scout's output before the Planner sees it. It produces two things:

  1. A ranked list of candidate hypotheses, each scored on three axes — testable · falsifiable · distinct — with a one-line rationale per axis. Distinct means "meaningfully different from the other candidates", not "rephrased".
  2. A Deep-Modules note for the Planner (Ousterhout bias): for the chosen approach, flags any component the Scout sketched that would be a shallow abstraction (mostly pass-through, high interface surface, low information-hiding) and suggests a deeper alternative.

The critic may also critique the spec itself: unresolved [NEEDS CLARIFICATION] markers prompt a hx_clarify recommendation; visibly thin FINER / PICOT / GQM prompts a Scout send-back recommendation.

This stage is also where the idea-tournament ranking lives for the rediscover loop (Workstream H). The ranking is written back to the hypothesis thread; subsequent iterations use the ranked order.

3. Planner model

The Planner reads the spec and the chosen approach (with the Scout Critic's Deep-Modules note) and writes a concrete validation plan with numeric bands. It is biased to design it twice:

  • For each non-trivial component, sketch two alternatives and pick the deeper one — high implementation behind a small interface, hides decisions callers don't need.
  • Reject shallow modules (thin pass-throughs, leaky abstractions, large interface for small payoff).
  • Record the rejected alternative + the reason in the alternatives field — that becomes the Rust-RFC Rationale & alternatives section of the report.

Output contract carries plan: {title, objective, steps, success_criteria, validation_bands, alternatives} + the standard atlas_writes + decision_card. The validation_bands is the contract for the deterministic Validator below.

4. Builder model

The Builder writes runnable code / analysis scripts / structured outputs as artifacts. Each artifact's name is a relative file path (e.g. src/sim.py, tests/test_x.py) sanitised by sandbox.sanitize_code_artifacts and written into the project's artifacts/ directory. Helix never executes the artifacts — they are written, not run.

The Builder also reports metrics that the Validator will check against the Planner's bands.

↳ TDD task loop (F.5)

The Builder advances one task at a time from the Planner's tasks.md (an ordered YAML list of {id, title, deps, acceptance}), test-first. Each hx_submit is one phase of one task:

stateDiagram-v2 [*] --> test : pick next task test --> impl : test phase submitted (failing test present) impl --> refactor : impl phase submitted (test goes green) refactor --> done : refactor phase submitted done --> [*] : pick the next task test --> test : send-back / revision (same phase, same task) impl --> impl : send-back / revision refactor --> refactor : send-back / revision

The orchestrator enforces the contract in core.agents._check_tdd_guards:

ViolationReject reason
Batched submission (task_ids list)"submit one task at a time"
Unknown phase"phase must be one of (test, impl, refactor)"
phase: "test" with no test file in artifacts"phase='test' requires at least one test artifact"
Phase jump (e.g. test → refactor)"expected phase 'impl' next"
Repeating a phasesame as above
Submit after refactor"task <id> is already done"
Test-disabling content (@pytest.mark.skip, @unittest.skip, pytest.skip(…), @pytest.mark.xfail(strict=False))"artifact … disables a test"

A violation cleanly surfaces as StageResult.error; the orchestrator pauses the run resumably and the agent fixes it on re-submit for the same task_id / phase. Legacy submits without a task_id / phase bypass the loop (the guardrail only activates when those fields are present), so the TDD discipline is opt-in per project.

5. Validator deterministic

The Validator is the only deterministic stage. It calls no model and costs no tokens. It checks each reported metric against the Planner's bands: 

for result in s.experiment_results:
    metric, value = result["metric"], result["value"]
    if metric in bands and value is not None:
        lo, hi = float(band["min"]), float(band["max"])
        if not (lo <= float(value) <= hi):
            flags.append(f"HARD: {metric}={value} outside band [{lo}, {hi}]")

Any HARD: flag routes the run back to the Builder with the flags as feedback (this is one of the deterministic iterate routes — it happens regardless of autonomy). A non-numeric value lodges as a SOFT: flag and the run continues. The Validator's own snapshot still runs through _write_report so there is a Validator HTML report with the hard-flag list.

6. Results Critic model

The Results Critic judges the outcome and returns a structured verdict: 

{
  "assessment": "…",
  "strengths": ["…"],
  "weaknesses": ["…"],
  "recommendations": ["…"],
  "verdict": "ship | iterate | abandon",
  …
}

A verdict: "iterate" is the second deterministic route back to Builder (the orchestrator picks this up regardless of autonomy). When artifacts contain code, the critic is instructed to reach for the security-review skill (graceful — use if installed). In the rediscover loop the critic additionally returns a verdict per hypothesis (supported · refuted · inconclusive) and a revised statement carried into the next iteration via the hypothesis thread.

Stage HTML reports

Each stage writes a self-contained HTML file at atlas/projects/<id>/reports/<stage>.html (overwritten each run; history lives in the snapshot DAG). The file is single-file by design — embedded JSON annotation block, inlined CSS + JS overlay, no external runtime — so it can travel by email or be opened from a USB drive.

StageTemplate shape
ScoutIMRaD + PRISMA flow
Scout CriticBlameless PIR with hypothesis rankings + Deep-Modules note
PlannerRust RFC (Motivation · Design · Rationale & alternatives)
BuilderMADR + PR-checklist
ValidatorIMRaD-Methods/Results with HARD-flag list
Results CriticBlameless PIR + verdict

The annotation overlay supports four researcher actions per critic note (Accept · Reject · Reply · Send back ↩) plus Add comment. Save the modified file in place (File System Access API on Chromium; Download fallback elsewhere), then ask the agent to send back the <stage> report — Helix runs hx_report_send_back, parses the embedded annotations, and threads the feedback into the next pass of that stage.

Constitution & spec

Two pieces of context every agent reads on every turn:

  • Constitution — workspace-level non-negotiables at <workspace>/constitution.md. Loaded by core.agents.load_agent and appended to every agent's system prompt. Per-project override at <project>/constitution.md wins if present. Edit it to set language / framework / testing / style / definition of "done". (Spec §F.1)
  • Spec — per-project at atlas/projects/<id>/spec.md. YAML frontmatter {finer, picot, gqm, datasets, open_questions, gate} + free-form markdown body. Scout owns initial authorship; hx_clarify walks unresolved markers; every later stage reads it. (Spec §B + F.2/F.3)

Authoring a custom agent

Override any stage for a project by dropping a markdown file at:

<project>/agents/<stage>.md

The shape:

---
name: scout                       # must match the stage name
order: 1                          # determines pipeline position
kind: llm                         # llm | deterministic
model_stage: scout                # ModelConfig routing key
atlas_write: true                 # may emit atlas_writes
snapshot: after                   # when the snapshot is minted
---
You are a research scout. [system prompt body]

Respond with JSON:
{ … your output contract … }

Three concrete rules:

  1. Reuse the Decision Card shape as the last field of your output. Otherwise the snapshot stores a generic card.
  2. Adopt the Workstream-G write protocol (action · because · provenance · spec_refs) when you emit atlas_writes. The sanitizer will compat-default missing fields, but the orphan-by-policy lint will flag unspec'd writes.
  3. If you change a context builder (core.agents._ctx_<stage>) or response mapper (_map_<stage>), keep the function pure on (state, ctx) → string for context, and on (response_json, state, ctx) → dict-of-updates for the map. Raising AgentInputError from the map cleanly produces a StageResult.error and the existing send-back path picks it up.

See Gates & control for how the Plan governs autonomy at each transition, and how send-back routes back through this pipeline.
