Helix

A collaborative research orchestrator and discovery engine, driven by a Claude Code agent over MCP.

Helix turns source material — papers, code, data — into a validated result through a disciplined six-stage pipeline, with a human checkpoint at every transition. It maintains a typed, frontmatter-driven research wiki (the Atlas) that compounds across projects, and a content-addressed snapshot DAG (the Snapshots) that makes every step inspectable, diffable, branchable, and resumable.

Helix has no model of its own and holds no credentials. The Claude Code agent is the model. Each stage hands its prompt back to the agent through the tool loop (hx_step / hx_submit); the server runs no sampling and stores no API keys.

What you get

Pipeline

Forge

Six stages with HITL at every transition. Each stage emits a Decision Card and a self-contained HTML report with an annotation overlay you can mark up and send back.

Wiki

Atlas

Markdown-on-disk pages with typed YAML frontmatter, a closed action vocabulary (ADD/UPDATE/SUPERSEDE/LINK/NOOP), mandatory provenance, spec cross-references, and an 11-check hygiene linter.

Version control

Snapshots

A content-addressed DAG of every transition. Branch, freeze, fork (single-file portable bundle), and resume from any point — even after a server restart.

Control

Gates

The run-scoped Plan controls per-gate autonomy. At a gate you can proceed, send back with a directive, or stop. Send-backs are themselves snapshots.

Discovery

Rediscover loop

The autonomous hypothesis loop runs across the existing six stages, capped at three iterations, carrying the hypothesis thread forward each pass.

Drive surface

MCP server

One stdio server. Curated tool set — no general code-exec. All filesystem writes pass the sandbox; project / run / bundle names go through validate_project_name.

90-second quickstart

pip install -e .                 # core
pip install -e '.[embed,pdf]'    # optional: semantic recall + PDF ingest

mkdir my-research && cd my-research
helix init                       # scaffold a workspace (once)
helix new bp-from-rppg           # scaffold a project
cd bp-from-rppg                  # open in Claude Code, then:

# ...in Claude Code chat:
> start helix

Read the Quickstart for the full walkthrough, or the Usage guide for control modes, send-back, observation, and resume.

Design principles

  • Agent-driven, no credentials. The server runs no model and holds no API keys. The agent answers every model-needing stage through the tool loop.
  • One snapshot per transition. The DAG is the source of truth for history. A snapshot calls no model.
  • One Atlas per workspace. The wiki is workspace-scoped so knowledge compounds across projects.
  • Closed action vocabulary. Every Atlas write declares ADD / UPDATE / SUPERSEDE / LINK / NOOP, a one-line rationale (because), and a spec cross-reference.
  • Sandbox-first. Every model-controlled string that reaches the filesystem passes a validator. Project / run / thread / bundle names go through validate_project_name; page and artifact content go through sanitize_atlas_writes / sanitize_code_artifacts.
  • Resumable everywhere. Budget pause, declined gate, lost client, server restart — all of them mint a snapshot and stop cleanly. hx_step resumes from the last snapshot.
