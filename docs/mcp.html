The MCP surface

One stdio MCP server (FastMCP). The client launches it via .mcp.json; there is no network service and no credential configuration.

Connection

helix init writes a server pinned to the interpreter Helix is installed in (sys.executable of the process running helix init): 

{ "mcpServers": { "helix": {
  "command": "/abs/path/to/python",
  "args":    ["-m", "helix.mcp.server"]
} } }

This is deliberate: the client resolves a bare command against its own PATH, which need not include the environment's bin/. Absolute interpreter + -m helix.mcp.server is PATH-independent and portable. helix-mcp (console entry point helix.mcp.server:main) and helix mcp run the same server by hand for debugging.

Agent-driven (no sampling)

Helix has no model of its own and stores no credentials. The client agent is the model. A model-needing stage does not sample: hx_step returns the stage's prompt as its tool output; the agent answers in its own turn; and hx_submit feeds that answer back in. See Technical architecture → Agent-driven for the responder seam.

The one remaining server→client callback — through the single seam in helix/io.py — is elicitation: structured questions to the user mid-tool (the hx_start wizard, every HITL gate, the hx_clarify walk, the hx_ask tool, confirmation before promoting to a published tier). Schemas are flat (string / enum / boolean / array), built by ask_text / ask_choice / ask_multi / ask_confirm. A client without elicitation cannot run gated tools, and fails fast with a clear message before any run starts (_missing_client_caps pre-flight).

Tools — pipeline

The pipeline is agent-driven: hx_step returns a stage's prompt; the agent answers; hx_submit advances. run_pipeline and hx_start are initialisers that return the first hx_step.

ToolPurpose
hx_step(folder, question?, autonomy_until?)Advance to the next stage that needs the model; returns its SYSTEM + USER prompt for the agent (or the final summary). First call initialises the run.
hx_submit(folder, stage, result_json, pending_token)Submit the agent's JSON answer for the pending stage; maps it, writes the Atlas, snapshots, runs the gate, and returns the next prompt. Guarded by pending_token.
hx_start(folder?)Guided setup wizard; creates the source folder if missing, then starts the run and returns the first stage's prompt.
run_pipeline(folder, question?, autonomy_until?)Start a run on a folder and return the first stage's prompt. autonomy_until ("", a stage name, "END", or "rediscover") auto-proceeds gates.
resume_pipeline(project, snapshot, at?, branch?, folder?)Rebuild a snapshot and re-enter at any stage / branch; folder= self-roots it.

Tools — spec / clarify / ask

ToolPurpose
hx_question_check(folder?, project?)Validate the project spec (FINER / PICOT / GQM filled; gate.status ready; no [NEEDS CLARIFICATION] / TODO / @open-question markers). Falls back to question.md if no spec yet. Spec §B + F.2/F.3.
hx_clarify(folder?, project?)Walk every unresolved clarification marker via elicitation; patch the spec in place with anchored SEARCH/REPLACE. Spec §F.3.
hx_ask(prompt, schema_json?, project?, field_name?)Ask the researcher a structured question via MCP elicitation. Default schema is a flat text input; pass a JSON schema for choice / multi / confirm. The Q&A is logged onto the pending step's clarifications list so the next stage's prompt surfaces it. Spec §C.
hx_constitution_init(folder?)Ensure the workspace has a Constitution (constitution.md) and return it. Idempotent. Spec §F.1.

Resources

Read-only, addressed by URI:

TemplateContents
atlas://{path}A page, with its frontmatter, by repo-relative path.
snapshot://{project}/{snap_id}A snapshot's metadata as JSON.
hot://{project}The project's hot-cache working state ("where were we?").
thread://{project}/{name}A project thread (hypothesis · data · spec · plan · design · code-org · glossary) — current state.
thread-at://{project}/{name}/{snap_id}The same thread, bi-temporal — state at the given snapshot.

Resource paths that include a project name are confined by validate_project_name (or its equivalent in spec_path / threads._resolve_path) before any file read — traversal attempts return a clean error rather than leaking file contents.

Safety posture

Three things together:

  1. Curated tool set, no general code-exec. The Builder writes artifacts; Helix never runs them. There is no arbitrary shell, no eval, no dynamic exec.
  2. Client consent. State-changing tools are confirmed by the client (the host's consent model) before they run; promotion to a published tier additionally elicits an explicit confirmation.
  3. Sandbox-first. All LLM-generated page and artifact content passes the sandbox (sanitize_atlas_writes / sanitize_code_artifacts) before it reaches disk. Model-controlled project / run / bundle / spec / thread names are confined by validate_project_name at every filesystem path root (no separators, no .., no leading dot). Atlas reads canonicalise paths via Atlas._safe_resolve as defence in depth.
Recent hardening
core/spec.spec_path and core/threads._resolve_path now call validate_project_name at the single point of path construction; the MCP tools (hx_question_check, hx_clarify) catch SandboxError and return a clean Rejected (sandbox) message rather than propagating an exception. Regression tests in tests/test_spec.py and tests/test_threads.py verify the thread:// resource and the spec tools reject traversal attempts cleanly.
