Framing

"The companies that make meaningful contributions to scientific progress won't just sell research copilots. They'll be AI-native discovery engines that work alongside researchers to propose and validate hypotheses."
— Y Combinator, Request for Startups

Helix is the whole system: a Forge (the six-stage agent-driven pipeline), an Atlas (a typed, content-addressed knowledge base that compounds across projects), and Snapshots (a content-addressed DAG of every transition with branch / freeze / fork / resume). Together they make Helix a collaborative research orchestrator and discovery engine — it works alongside the researcher to propose and validate hypotheses, surface contradictions and gaps, and produce validated, reproducible artifacts.

The shape: data is a first-class concern at every stage (Workstream E). The pipeline is threaded — a small set of longitudinal artifacts (question/hypothesis, data, spec, plan, design, code organization, glossary) is created early and updated by every stage (Workstream E). Every stage emits a self-contained HTML report with an inline annotation overlay for the Scout Critic's notes and the researcher's send-back marks (Workstream D). The pipeline runs as disciplined agentic engineering — Constitution → spec → clarify → plan → tasks → implement → verify, with a Ubiquitous Language, TDD per task, and a Deep-Modules design bias (Workstream F). At the heart of it all sits an autonomous hypothesis loop (Workstream H) — propose → validate → revise — borrowed from the agentic-discovery literature (Sakana AI Scientist-v2, Google's AI co-scientist, FunSearch, Coscientist). The Atlas (Workstream G) is the cross-project knowledge base these artifacts feed into and discoveries accrete to.

Goals · Non-goals

Goals

  • Restore the Atlas as a true cross-project knowledge base (Workstream A + G).
  • Two-command lifecycle: helix init (workspace, once) + helix new <project> (per project).
  • Guided Scout with an "interview the researcher" loop and an explicit completeness gate; a skip-to-full-Scout always available.
  • Treat datasets as first-class artifacts: scout, plan, build, validate (Workstream E).
  • Make persistent artifacts ("threads") explicit and longitudinal (Workstream E).
  • One critic only (Scout Critic) — at the spec gate, where errors are cheapest to fix.
  • Every stage's report is a self-contained HTML with an embedded annotation overlay; round-trip is single-file.
  • Define the Atlas write protocol (actions + dedup gate + provenance) as part of the full Atlas spec.
  • Run the pipeline as spec-driven agentic engineering (Constitution, Clarify loop, tasks, TDD, Deep Modules, Ubiquitous Language).
  • Integrate Claude skills/plugins per their licenses.

Non-goals

  • No return to server-side sampling or credentials.
  • No change to the sandbox model or the dependency-light core.
  • No automatic migration of data created since the self-rooting change.

Architecture at a glance

Two roots: a shared workspace (the Atlas + the Constitution) and a per-project run root (snapshots / runs / pending / source folder). The agent is the model.

Forge with one critic

Six stages with a single critic — after Scout, where errors are cheapest to fix (the spec is the source of truth; spec drift downstream is far more expensive than spec drift caught at scout-gate). The Validator is deterministic (no critic — it is the gate). Results Critic returns a verdict at the end (not a per-stage critique).

flowchart LR S[Scout] --> SC[Scout Critic] SC --> P[Planner] P --> B[Builder · TDD task loop] B --> V[Validator · deterministic] V --> R[Results Critic · verdict] R --> END[done · iterate · send-back] classDef crit fill:#fff8c5,stroke:#bf8700,color:#573a08; classDef det fill:#dafbe1,stroke:#1a7f37,color:#1a7f37; class SC,R crit class V det

Why only one stage critic? The literature on critique loops (Self-Refine, Reflexion, Constitutional AI) is consistent that per-step critique inflates cost and risks loops on long-horizon tasks. With Helix's spec-as-source-of-truth discipline, the leverage point is at the spec — once the spec passes scout-critic review, downstream errors mostly become spec drift that surfaces in the Validator + Results Critic, where the existing send-back path handles them. References: Self-Refine, Reflexion, Constitutional AI.

Workstream A — Workspace & init new

Problem

Today helix init <name> creates a per-project folder with its own helix.toml and atlas/, and the agent-driven use_root(folder) change bound the Atlas to that folder — so the Atlas became per-project (an isolated wiki per research project), breaking cross-project compounding. The fix is structural: split the init lifecycle in two.

Workstream B — Guided Scout: interview the researcher expanded

Scout is a sequence of standardized sub-phases. The discipline is borrowed from "interview the developer" (Maximiliano Contieri's spec-driven dev pattern — substack) and Decision-Oriented Programming with Aporia (arXiv:2604.05203, which reports ~5× / ~61% relative reduction in mental-model/implementation mismatch when decisions are elicited up front). For Helix's research context this becomes "interview the researcher": structured Q&A with a named question set before any model work begins.

Sub-phase requirements (agent-parseable)

Each requirement is a single sentence the Scout prompt must satisfy. Numeric IDs are stable so the spec and the prompt can reference them.

Skip-to-full-Scout (always available)

REQ-B-SKIP-01   At the start of any sub-phase, the agent MUST offer the researcher a "skip the interview and run full Scout autonomously" option, and MUST proceed only after a positive confirmation via hx_ask (auditable). The skip is not auto-chosen even on autonomy_until: scout; the researcher's intent is required.
REQ-B-SKIP-02   If skipped, Scout runs end-to-end without the Frame / Source / Synthesize sub-phase questions, producing the spec, source-page writes, and the candidate-hypotheses thread in a single submit. The Scout Critic still runs at the gate.

Frame — interview the researcher

REQ-B-FRAME-01   If question.md is empty or sparse, the agent MUST conduct a multi-perspective elicitation (STORM-style perspective prompts) via hx_ask before any literature work.
REQ-B-FRAME-02   The sharpened question.md MUST carry a typed YAML frontmatter scoring three canonical research-question frameworks: FINER (Feasible · Interesting · Novel · Ethical · Relevant), PICOT (Population · Intervention · Comparator · Outcome · Time), and GQM (Goal → Questions → Metrics).
REQ-B-FRAME-03   Every unresolved point MUST be marked in the body with the literal placeholder [NEEDS CLARIFICATION: …] (Spec Kit pattern). The agent MUST NOT advance past Frame while any marker remains.
REQ-B-FRAME-04   Each elicitation MUST be a single named question (free-form chat is not auditable); answers MUST be persisted as part of the Decision Card and the snapshot.

Data scout (J → here, see Workstream E)

REQ-B-DATA-01   The Frame interview MUST include at least one named question per candidate dataset: source · license · size · format · access · suitability · limitations. Each candidate dataset MUST be recorded as an Atlas entity page (entities/datasets/<slug>.md).

Completeness gate (the named contract)

REQ-B-GATE-01   A new tool hx_question_check(folder) MUST validate the question.md frontmatter against the schema below and refuse to advance Scout while any field is unresolved or any [NEEDS CLARIFICATION] remains in the body.
REQ-B-GATE-02   The gate-pass MUST be confirmed by the researcher via hx_ask elicitation; gate.by and gate.at MUST be filled in the frontmatter and travel with the snapshot.
REQ-B-GATE-03   The Clarify loop (Workstream F · K.3) MUST be the recovery path on gate failure: one open marker resolved per turn via hx_ask, then re-check.
---  # projects/<id>/question.md frontmatter (the gate's contract)
finer:          { feasible, interesting, novel, ethical, relevant }   # 1-sentence rationale or "n/a: <reason>"
picot:          { population, intervention, comparator, outcome, time }
gqm:            [{ goal, questions: [...], metrics: [...] }]           # ≥ 1 tree
datasets:       [{ slug, source, license, size, format, access, suitability, limitations }]
open_questions: []                                                     # ALL [NEEDS CLARIFICATION] markers resolved
gate:           { passed: false, by: "", at: "" }
---

Source · Literature dive · Synthesize · Spec

REQ-B-SRC-01   Gated on Frame's pass. The agent MUST confirm the inbox contents and any researcher-provided sources via hx_ask before the literature dive.
REQ-B-LIT-01   The agent MUST use its own web tools (WebSearch · WebFetch) plus the Atlas to do a real survey. Results MUST land as typed Atlas pages with an Elicit-style structured-extraction frontmatter (claim · evidence · method · sample · limitations — Elicit pattern).
REQ-B-SYN-01   The agent MUST identify consensus, contradictions and gaps across sources + the web, and record each as an Atlas page (per Workstream G's action vocabulary).
REQ-B-SPEC-01   The agent MUST emit projects/<id>/spec.md (Workstream F): sharpened question, scope, success criteria + validation bands, constraints, known contradictions/gaps, dataset requirements (E), candidate hypotheses (Workstream H), and the definition of value/done.

The whole loop is exploratory → narrow: each sub-phase can iterate with the researcher before advancing. Implementation is prompt-level (the Scout agent gains the sub-phase checklist + the elicitation tools), plus a small spec-file writer and hx_question_check.

Open decisions

B1 Number of standard interview questions per sub-phase: a tight 6-question minimum (recommended) vs. an unbounded "until satisfied" loop. B2 Skip-to-full confirmation: single hx_ask (recommended) vs. typed phrase ("yes, skip") to avoid accidental skip.

Workstream C — Interactive agents (ask the user anywhere)

Every stage prompt explicitly permits the agent to ask the researcher for clarification before submitting. A thin tool hx_ask(question, schema?) round-trips a structured question through the existing io.elicit seam and records the Q&A in the snapshot, so important clarifications are auditable.

Both modes ship: free chat-level questions for casual ambiguity; hx_ask for anything that should be recorded in the run history. Answers refine the appropriate threads (Workstream E) and appear in the stage report (Workstream D). Workstream B's Frame interview is the canonical example of named-question-set elicitation.

Open decision

C4 Interactive clarification: prompt-only chat vs. always-recorded hx_ask. Recommendation: both; hx_ask for anything that should be recorded.

Workstream D — HTML reports + critic overlay + send-back consolidated · supersedes old I

Each stage emits a self-contained HTML report at projects/<id>/reports/<stage>.html, overwritten on every (re-)run. The single critic (Scout Critic) — and any researcher annotations — live inside the HTML file itself, as embedded JSON read and written by a small inline JS overlay. The HTML is the round-trip unit: one file in, one file back.

The "single HTML file is the unit" tech

ConcernHow it's done — with just HTML
Where the data livesInside the HTML, in a <script type="application/json" id="hx-annotations">…</script> block. Conforms to the W3C Web Annotation Data Model (W3C TR): each annotation has a body, a target with FragmentSelector + TextQuoteSelector + TextPositionSelector (Hypothes.is-style fuzzy anchoring), plus a Helix-specific state (open · accepted · rejected · replied) and the actor/timestamp log.
How the overlay reads/writesA tiny self-contained JS bundle (inlined by the renderer) reads the JSON on load, paints the side-notes (Distill-style — Distill), and on every action updates the JSON in the DOM and on disk via the File System Access API (window.showSaveFilePicker · Chromium browsers). Non-Chromium fallback: a Download annotated copy button writes a fresh HTML the researcher drops back in the project folder.
Researcher actions per annotationAccept, Reject (with optional reason), Reply (Google-Docs-style threaded comment — Docs comments help), Send back to stage (the new one-click send-back). All resolution events go into the in-page log.
Researcher's own comments / suggested editsSelecting any text in the report exposes Add comment and Suggest edit. The latter materialises as a CriticMarkup patch ({++add++} / {--del--} / {~~old~>new~~}) — Google Docs' Suggesting mode applied to a static HTML target.
How the gate is presentedThe gate prompt the researcher sees in the chat is the report's Summary + the count of unresolved annotations + a clickable file:///<abs-path>/reports/<stage>.html link (most modern terminals + Claude Code render file:// URLs as clickable). For non-clickable terminals a "Open report" command is suggested.
What the next-run agent seeshx_report_send_back(project, stage) parses the modified HTML, extracts the embedded JSON, and emits a goto <stage> with a structured feedback bundle: kept critic annotations (still-open items the researcher chose not to reject) · rejected critic annotations + their reasons (so the agent doesn't reintroduce them) · replies on critic annotations · researcher's own comments + suggested edits · the previous report's HTML as context. The stage re-renders the report and the cycle continues.

Per-agent HTML templates (industry-shaped, with diagrams where appropriate)

Each template is the smallest section set the surveyed literature treats as the standard for that kind of document. Fixed top-level sections so the reader skip-scans and an editor knows exactly where to add content. The HTML rendering adds capabilities markdown can't carry: anchored TOC + stable section IDs, Distill-style side-notes, sortable/filterable tables, citation tooltips, confidence/verdict badges, a small mermaid pipeline strip showing the stage's place in the run, dark mode and a print stylesheet.

One canonical report per stage; lint enforces

Each report writes to projects/<id>/reports/<stage>.html and is overwritten on every (re-)run of that stage. History is the snapshot DAG; the Atlas page is the current view. The Decision Card stays the structured backbone — present in every snapshot — and a renderer projects it into the per-agent template, then emits self-contained HTML (Quarto's embed-resources: true pattern — Quarto; Pandoc --embed-resources). Lint adds a stale-report check: any file under projects/<id>/reports/ that doesn't match the canonical <stage>.html name, or whose stage didn't run on the latest snapshot path, is flagged with a one-click fix to delete or re-render.

Round-trip patterns (unchanged from v4)

  • Frontmatter-driven form for structured fields (status · threads touched · verdict · links · spec refs) — pattern after Astro content collections (Astro) and Obsidian Bases (Bases).
  • [NEEDS CLARIFICATION: …] as literal in-text markers — Spec Kit's pattern (spec-driven.md).
  • Stable anchored headings; agents edit one section without touching others.
  • Anchored SEARCH/REPLACE block edits for agent-authored amendments — Aider's format (Aider edit formats).
  • Supertag-style stamps, not bindings (Tana supertags · Notion DB templates).
  • AGENTS.md as the per-stage agent-context surface — Linux Foundation Agentic AI Foundation open standard (agents.md, GitHub blog).
  • Sandboxed-diff round-trips — Plandex's "cumulative diff review sandbox" (plandex); Helix already snapshots per stage.

Workstream E — Threads (incl. data as a first-class thread) consolidated from H + J

A thread is a persistent, evolving artifact that spans the pipeline. Each thread has a typed home in the Atlas, a current state, a history (via snapshots), and a small status machine (Proposed → Accepted → Superseded). Stages contribute updates to relevant threads; the report (Workstream D) declares which threads were touched. The pattern is borrowed from ADRs as living documents, design-rationale / IBIS (Wikipedia), and Graphiti's temporally-versioned facts (arXiv:2501.13956).

Data as a first-class thread

Across the surveyed agentic-research systems (STORM, Sakana AI Scientist-v2 — arXiv:2504.08066, Elicit, OpenAI Deep Research — openai.com), datasets are treated as fixed inputs. For computational research that is a hole. Helix makes data a first-class artifact, tied to the data thread, with per-stage instructions:

StageData-aware sub-instruction
ScoutData scout: identify candidate datasets; record each as entities/datasets/<slug>.md with {source, license, size, format, access, suitability, limitations}; flag access constraints.
Scout CriticCritique data choice vs. the question; flag unsuitable datasets in the overlay.
PlannerData plan: which datasets, preprocessing, splits, ground-truth, validation bands tied to data.
BuilderData build: scripts that fetch/transform/version the data; the built/curated dataset is a versioned artifact with a hash; partial datasets are snapshots on the data thread.
ValidatorDeterministic checks against the data spec (schema, splits respected, bands met).
Results CriticGeneralization, leakage, fairness checks tied to the data thread.

Lint adds dataset-without-license and dataset-without-source.

Open decision

E1 Thread schema (front matter + status machine). Recommendation: each thread is an Atlas page with front matter {thread, status, opened_at, last_touched_at, contributors} + a body of sectioned per-snapshot updates.

Workstream F — Agentic engineering (incl. spec-driven) consolidated from E + K

The pipeline runs as a disciplined agentic-engineering practice: a Constitution establishes non-negotiables; a Spec → Clarify → Plan → Tasks → Implement → Verify loop replaces ad-hoc prompting; a Ubiquitous Language keeps every stage aligned; TDD paces the Builder; a Deep-Modules design bias keeps the system from sprawling. This is the synthesis of DDD, TDD, APoSD + GitHub's Spec Kit flow.

Spec Kit's commands, mapped to Helix stages

flowchart LR CO[/constitution/
hx_constitution_init at helix init] --> SP[/specify/
Scout · Spec sub-phase] SP --> CL[/clarify/
hx_clarify · loop · gates the next phase] CL --> PL[/plan/

Validator + Results Critic vs. spec]

The discipline is "spec is the source of truth, not the code" — when drift is detected mid-Builder, the run goes back (existing send-back) to amend the spec first, and the drift itself becomes an explicit follow-up task appended to tasks.md (not a hand-wave or an in-place patch). References: Spec Kit, Fowler — Understanding SDD, Contieri, Aider architect/editor split.

F.1 — Constitution (workspace-level non-negotiables)

Before any stage runs, a Constitution is established as a workspace-level artifact <workspace>/constitution.md, with an optional per-project override at projects/<id>/constitution.md. It captures the project's non-negotiables: language(s), framework, testing approach, architectural style, definition of done. Pattern after Spec Kit's memory/constitution.md. The Constitution is part of every agent's prompt context; stages re-read it at the start of their turn. hx_constitution_init runs at helix init (Workstream A) and elicits the small set of choices that populate the file; later edits are normal markdown.

F.2 — Spec (the source of truth)

Scout emits projects/<id>/spec.md (Workstream B). Planner derives the plan from it, Builder builds to it, Validator + Results Critic validate against it. Spec changes are a normal send-back. The spec contents match the canonical list — goals, scope, constraints, acceptance criteria, validation (Osmani — How to write a good spec for AI agents). Helix's own development is also spec-driven: this document is 0001 in docs/specs/; numbered, append-only, written before code.

F.3 — Clarify loop

hx_clarify(project) scans the spec and any thread the spec depends on for [NEEDS CLARIFICATION: …] markers and other ambiguity signals, asks the researcher one at a time via hx_ask (Workstream C), and patches the spec in place with anchored SEARCH/REPLACE edits. The spec advances only when no markers remain. This operationalises the Decision-Oriented Programming with Aporia finding — arXiv:2604.05203 — that structured up-front decision elicitation reduces mental-model/implementation mismatch by ~5× (~61% relative).

F.4 — Ubiquitous Language (the glossary thread)

A Ubiquitous Language (Evans, DDD — Fowler bliki, DDD Reference PDF, Agile Alliance, InfoQ — DDD Quickly) is a single shared vocabulary used by domain, spec, plan, and code. In Helix it is realised as a thread (E): a glossary page maintained at projects/<id>/threads/glossary.md with one term per heading, a one-sentence definition, aliases, and links to the concepts/sources that define it. Workspace-level shared terms live at <workspace>/atlas/concepts/glossary/<term>.md. Scout seeds the glossary from the literature dive; every later stage's prompt includes it; lint adds vocabulary-drift for terms used in spec/plan/code not in the glossary.

F.5 — TDD as the Builder's task loop

The Builder runs one task from tasks.md at a time, in a red / green / refactor cycle aligned to Kent Beck's "Augmented Coding" (tidyfirst, Pragmatic Engineer interview, Engineering Harmony — How TDD accelerates AI coding). Per task: pick → write the failing test first → implement just enough to pass → refactor → snapshot per task. Guardrails: per-task iteration cap deferring to the budget ceiling; the Builder refuses submissions that disable tests, skip the test-first step, or batch multiple tasks; autonomy_until relaxes interactivity but not the test-first invariant.

F.6 — Deep-Modules design bias

Both the Planner's prompt and the Builder's self-check rubric prefer deep modules: small public interfaces hiding substantial implementations (Ousterhout, A Philosophy of Software Design — book, Pragmatic Engineer interview). Apply design it twice: produce two structurally distinct plans for the same spec and choose between them with explicit trade-offs (fits naturally in the Rust-RFC Rationale & alternatives section, D).

Open decisions

F-1 Constitution location: workspace-level (recommended; matches Spec Kit) vs. per-project only. F-2 /clarify hard gate (recommended) vs. advisory. F-3 tasks.md generated by a Planner sub-phase (recommended) vs. its own stage. F-4 Glossary thread: workspace + project (recommended) vs. project-only. F-5 TDD strictness: hard by default (recommended) with explicit Constitution opt-out vs. soft. F-6 Spec gate: advisory (recommended; reports declare advancement) vs. enforced hard gate.

Workstream G — Atlas: full technical spec (incl. write protocol) consolidated

The Atlas is Helix's typed, content-addressed wiki + knowledge graph that compounds across research projects. Atomic markdown pages (Andy Matuschak — Evergreen notes should be atomic); wikilinks + backlinks (Obsidian — Properties, Bases); tier-based curation (Zettelkasten); closed-set write protocol (mem0 ADD/UPDATE/DELETE/NOOP — arXiv:2504.19413; Letta core-memory blocks — letta.com; Zep/Graphiti bi-temporal graph with edge invalidation — arXiv:2501.13956). Helix-specific: a multi-kind lint, a sandboxed write path, a hot cache for "where were we?" recovery.

Page model

Every page is markdown with typed YAML frontmatter:

---
id:         atlas:<type>:<slug>
type:       source | concept | entity | report | thread | spec | plan
tier:       working | active | canonical | published
title:      "..."
summary:    "one-line"
aliases:    ["..."]
links:      ["atlas:concepts:slug", "atlas:sources:slug"]
provenance: { stage, run_id, snapshot_id, sources: [...], confidence }
ts:         { created_at, updated_at }
---

Frontmatter is queryable (Obsidian Bases pattern). Bodies are atomic — one concept per page (Matuschak). Wikilinks ([[...]]) create backlinks automatically (Obsidian / Roam); the graph view falls out of the link store. Code artifacts are content-addressed under <project>/.helix/snapshots/objects/<sha>; pages reference by hash so identical files across iterations store once.

Tier model + promotion

TierMeaningPromotion
workingthe default — any stage write lands hereautomatic
activereviewed inside this projectmanual or via hx_atlas_reconcile
canonicalreviewed cross-project; the workspace's "what we know"hx_atlas_promote + elicitation confirm
publishedexternally shareable; immutable for citationhx_atlas_promote + elicit + hx_snap_freeze

Capture is cheap (everything lands at working); curation is gated (Zettelkasten fleeting → permanent).

Recall (auto-routing)

flowchart LR Q[hx_atlas_recall query] --> R{mode} R -- auto / lexical --> L[Lexical · full-text + alias] R -- semantic --> S[Semantic · fastembed ANN] R -- graph --> G[Graph · k-hop on wikilinks] R -- community --> C[Community · label propagation] L --> OUT[References only:
id · title · tier · summary · score] S --> OUT G --> OUT C --> OUT OUT -- hx_atlas_get --> BODY[Page body]
  • lexical — full-text + alias match; the always-available fallback.
  • semantic — fastembed (optional) over body sha256; vector ANN; lazy-imported per the dependency-light invariant.
  • graph — k-hop traversal over the wikilink graph from a seed page.
  • community — label-propagation communities; returns the seed's community + a synthesis page if one exists.

Recall returns references only; hx_atlas_get fetches a body. The find/fetch split keeps a search from flooding the model's context.

Ingest

hx_atlas_ingest(path) processes atlas/inbox/ (or one file). It is idempotent: a .manifest.json keyed by sha256 records what has been processed. New files become type: source pages; originals move to atlas/raw/. Markdown/text natively; PDF via the optional pdf extra (pymupdf); other formats land as raw/ with a thin stub page.

Write protocol (action vocabulary + dedup gate)

Every Atlas write declares one of:

ActionWhen
ADDa new page; no existing page covers this exact concept
UPDATEan existing page is the right home; merge new content
SUPERSEDEa prior page is now wrong; mark superseded, link forward
LINKno content change; add a relationship edge between existing pages
NOOPnothing belongs in the Atlas — it lives only in the snapshot

Each write carries a one-line because rationale. Mirrors mem0's ADD/UPDATE/DELETE/NOOP (arXiv:2504.19413, mem0 docs) and Letta's memory_insert / memory_replace / memory_rethink on core-memory blocks (Letta docs).

Dedup gate — hx_atlas_propose(write) runs the existing auto-routing recall against the page's title/summary/aliases, returns the top-3 candidates + a recommended action. Agents call this before an ADD. Mirrors Graphiti's hybrid-search entity-resolution + edge-dedup at write time (arXiv:2501.13956) — composing the recall we already have rather than building new infrastructure.
Provenance required — every page carries {stage, run_id, snapshot_id, sources: [...], confidence}. Most is derivable in the sanitizer; missing provenance fails validation.
Capture cheap, curate gated — writes land at tier: working by default (Zettelkasten — zettelkasten.de). A periodic hx_atlas_reconcile re-runs lint, dedups, proposes promotions.
Spec-driven cross-reference — every write declares which spec line(s) it supports / contradicts / extends. Pages citing no spec line are flagged by lint as orphan-by-policy.

Workstream H — Discovery engine: propose · validate · revise renamed from L

Helix's pipeline IS the discovery engine. The autonomous hypothesis loop — propose → validate → revise — runs end-to-end across the existing six stages, not as a new stage. A new hypothesis thread (E) holds the candidates and their status.

flowchart TB Q[Question / hypothesis thread] --> SC[Scout · propose hypotheses + gaps] SC --> SCC[Scout Critic · rank · idea tournament] SCC --> PL[Planner · design validation plan] PL --> BU[Builder · execute under TDD] BU --> VA[Validator · programmatic checks] VA --> RC[Results Critic · automated reviewer · verdict] RC -- supported · refuted · inconclusive --> Q classDef new fill:#ddf4ff,stroke:#0550ae,color:#0550ae; class Q,RC new

H.1 — A first-class hypothesis thread

Each entry carries {id, statement, type, support, refutations, status, last_touched_at, provenance}, where status ∈ {proposed, validating, supported, refuted, revised, retired}.

H.2 — Per-stage roles

StageDiscovery-engine role
Scout (B)Propose candidate hypotheses from the literature dive + the question; surface gaps and contradictions that imply hypotheses worth testing.
Scout CriticScore candidates: testable · falsifiable · distinct. Idea-tournament / ranking structure from Google's AI co-scientist (blog, arXiv:2502.18864) — Elo-style ranking over the candidates.
PlannerDesign the validation plan for the top-ranked hypothesis: data, methods, validation bands.
BuilderExecute under TDD (F.5). Agentic tree search / experiment manager from Sakana AI Scientist-v2 (arXiv:2504.08066, SakanaAI/AI-Scientist-v2).
ValidatorProgrammatic checks against the validation bands. FunSearch's programmatic validator discipline — execute the program, don't just judge the text (DeepMind blog, Nature 2023).
Results CriticAutomated reviewer: per hypothesis returns supported / refuted / inconclusive, plus a revised hypothesis for the next loop. Verdict pattern from Sakana.

H.3 — Closed vs. open loop

By default the loop is open: Results Critic returns its verdict and the run pauses for the researcher to decide whether to revise (send-back). A Plan mode autonomy_until: rediscover runs the loop closed for N iterations against the same spec — Coscientist's autonomous wet-lab framing (Boiko et al., Nature 2023).

H.4 — Safety + reproducibility

  • The hypothesis thread cites Atlas pages and dataset versions for every support/refutation (provenance per G).
  • Validator failures pause the loop; the loop never auto-promotes a hypothesis past supported without an explicit human elicit (mirroring hx_atlas_promote's gate).
  • A discovery-log lint check flags hypotheses that gained supported status without a validator pass.

Open decisions

H-1 Hypothesis-thread schema (severity bands; "distinct" metric for the idea tournament). H-2 Default iteration cap for the closed loop (recommendation: 3; capped further by the existing token/call budget). H-3 Ranking sub-agent vs. function of the Scout Critic (recommendation: function of Scout Critic — only one critic).

Risks

  • A regresses self-rooting if done carelessly. Keep operational state per-run; only the Atlas moves to the workspace.
  • Single critic is too thin if the spec misses something the Builder needs. Mitigated by F.3's Clarify loop + the Validator's deterministic checks + Results Critic's verdict.
  • Single-file HTML round-trip depends on the File System Access API in Chromium browsers. Non-Chromium fallback is download-and-replace; document it; consider a tiny local helper if it becomes a friction point.
  • Web nondeterminism in Scout's lit dive. Outputs must still be snapshot- and spec-anchored.
  • License variance. Re-verify per-skill before redistribution; never bundle anything beyond the explicitly-licensed subset.
