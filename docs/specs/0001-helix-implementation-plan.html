Spec 0001 — Implementation Plan & Design

Plan · v1 · Draft A concrete, file-level technical plan for shipping every workstream in spec 0001 v6 into the current codebase.

Approach

The plan is shaped by three constraints already true of the codebase:

  • Dependency-light core — helix.core, helix.io, and helix.orchestrator.loop import without mcp or fastembed. Anything in this plan that touches the SDK lives in helix/mcp/.
  • One router · one runner — all routing flows through core.transitions.next_stage; the only runner is loop.advance, re-entered per hx_submit. New workstreams compose; they never duplicate the router or the runner.
  • Snapshots never call a model · sandbox-first — every new write path passes through sandbox.sanitize_* / validate_project_name; new artifacts are content-addressed where they're bytes.

The work ships in eleven phases (mirroring spec §Phasing). Each phase is independently valuable; full pytest stays green between phases. The autonomous app.run harness (driven by the patched call_llm_json in conftest.py) is the carrier test surface for core changes; the in-memory MCP client (mcp.shared.memory.create_connected_server_and_client_session) is the carrier for end-to-end agent-driven tests.

Workstream B — Guided Scout: interview the researcher XL

Spec §B Phase 6 (with F.2/F.3) Depends on: D · E · F.2/F.3 · C

Goal

Scout becomes a sequence of standardized sub-phases (Frame · Source · Lit dive · Synthesize · Spec) — implemented at the prompt level inside the existing single Scout stage, plus a small completeness-gate tool. The agent uses hx_ask for structured questions; a skip-to-full-Scout option is always offered first.

Design note: where the sub-phases live

The single-stage shape is preserved. Scout's turn (one hx_step → many hx_ask elicits in conversation → one hx_submit) walks the sub-phases. The completeness gate is a hard server-side validator; the agent cannot submit a Scout result whose question.md frontmatter is incomplete.

Open implementation questions

  1. Where does the overlay JS bundle source live? Recommendation: a single hand-authored src/helix/core/reports/overlay.js + overlay.css read at render time and inlined (no build step). Alternative: a tiny TypeScript build under js/ with esbuild — only if the JS grows.
  2. How is the file:// link rendered in non-clickable terminals? Recommendation: print the URL on its own line so terminals that auto-link still recognise it; document a helix open <project> --report <stage> CLI alternative.
  3. Per-task snapshots in F.5: does each test-first / impl / refactor get its own snapshot, or only the green tasks? Recommendation: only green tasks (one snapshot per completed task) — the red phase is a fast-failing iteration inside the Builder's turn.
  4. The Scout Critic doing the ranking and the critique — does it confuse the rubric? Recommendation: a single prompt with two named sections (Rank · Critique) — the Critic's HTML report puts ranking in What I found, critique in Recommendations.
  5. Migration story for atlases created since the A regression: document a recipe; no auto-migration (per A6 recommendation). Open question: should helix new detect a pre-existing atlas/ in the project folder and offer to move it to the workspace?
