Quickstart

From pip install to a paused pipeline gate in five minutes.

Prerequisites

  • Python ≥ 3.11.
  • An MCP client whose agent drives the tool loop and supports MCP elicitation for gates. Today that is Claude Code.
  • No API keys. No model configuration. The agent is the model.

Install

pip install -e .                 # core (click · pyyaml · mcp · anyio · python-dotenv)
pip install -e '.[dev]'          # + pytest
pip install -e '.[embed,pdf]'    # optional: semantic recall + PDF ingest

Helix declares no dependency on an LLM SDK. The two optional extras are fastembed (local embedding model for semantic Atlas recall) and pymupdf (PDF text extraction during ingest). Skip them to keep the install minimal.

Scaffold a workspace

Helix is organised as one workspace, many projects. The workspace owns the shared Atlas (cross-project knowledge base); projects live as subfolders. Run helix init once per workspace; helix new <project> per project. 

mkdir my-research && cd my-research
helix init
# Workspace ready: /Users/you/my-research
#   Atlas at: atlas/
#   Next:     helix new <project-name>

This writes:

FilePurpose
helix.tomlWorkspace marker ([workspace]) + Atlas path ([atlas]).
.mcp.jsonRegisters the helix MCP server, pinned to the interpreter that helix init ran in.
constitution.mdNon-negotiables every agent reads (language / framework / testing / style / "done"). Edit it.
AGENTS.mdRecommended Claude Code skills (graceful — use if installed).
atlas/The shared Atlas tree: inbox/, raw/, sources/, concepts/ (incl. glossary/, methods/, contradictions/), entities/ (incl. datasets/), projects/.
README.mdWorkspace README explaining the layout.

Create a project

helix new bp-from-rppg
cd bp-from-rppg
# Project ready: /Users/you/my-research/bp-from-rppg
#   1. Edit  bp-from-rppg/question.md
#   2. Add   source files to bp-from-rppg/
#   3. cd    bp-from-rppg  && open in Claude Code, then: start helix

This writes, inside the project folder:

FilePurpose
question.mdYour research brief. Edit it before the first run.
helix.tomlPer-project [limits] (token_cap, call_cap).
.mcp.jsonSo Claude Code finds the server when it opens this folder.
CLAUDE.mdInstructions the agent reads on "start helix".

Add your source material (PDFs, papers, code, data) into this folder — anywhere. Or drop it into ../atlas/inbox/ for managed ingestion via hx_atlas_ingest.

First run

Open the project folder in Claude Code. It launches the MCP server automatically (via .mcp.json). Then talk in plain language. The client calls Helix's tools; the agent answers each stage's prompt; the run advances. 

you>   start helix
helix> What should we call this project?  (lowercase / digits / hyphens)
you>   bp-from-rppg
helix> One-sentence research question / description?
you>   Cuffless BP from smartphone rPPG video.
helix> How should it run?
       1 step-by-step (ask at every gate)
       2 auto up to a stage, then ask
       3 fully autonomous
you>   1
helix> Source folder to ingest (path)
you>   ./papers
helix> Started 'bp-from-rppg' (step-by-step).
       NEEDS MODEL — project 'bp-from-rppg', stage 'scout'.
       You are the model for this stage. Read SYSTEM + USER below, then …

What you've just seen is the hx_start wizard handing off to hx_step. The agent now reads the Scout prompt, produces the JSON the contract requires, and calls hx_submit — and the run advances to the next gate.

What happens at a gate

The full step → submit → gate → transition cycle:

sequenceDiagram participant U as You participant A as Agent (Claude Code) participant H as helix-mcp participant C as core (loop · stages · gates) U->>A: start helix A->>H: hx_start(folder) H->>U: elicit name / question / mode / sources U-->>H: answers H->>C: loop.advance — render Scout prompt C-->>H: NEEDS MODEL (Scout SYSTEM + USER) H-->>A: tool result: prompt + pending_token A->>A: reason, produce Scout JSON A->>H: hx_submit(stage='scout', result_json, pending_token) H->>C: inject answer · sanitize_atlas_writes · Atlas write · snapshot C->>H: gate decision needed H->>U: elicit — proceed / send back / stop? U-->>H: proceed H->>C: transition → next stage (Scout Critic) H-->>A: NEEDS MODEL: scout_critic prompt

At a gate you can:

  • proceed — go to the next stage in order.
  • send back — pick any earlier stage and write a directive; the directive is threaded into that stage when it re-runs.
  • stop — pause the run resumably.

Every transition (including a send-back) is a snapshot. The run is resumable from any of them. Read the Gates & control doc for the Plan, autonomy modes (auto_until, rediscover), and how deterministic routing (Validator hard miss, Results-Critic iterate) interacts with HITL.

Next steps

  • Drop a PDF in atlas/inbox/ and ask Claude Code to "process my inbox". It runs hx_atlas_ingest — idempotent, sha256-keyed; the original moves to atlas/raw/ and a typed source page lands under atlas/sources/.
  • Ask "what do we know about X?". The agent calls hx_atlas_recall — auto-routing across lexical, semantic, graph, and community modes — and surfaces references only (bodies fetched on demand).
  • After a few runs, ask "health-check the atlas". Helix runs the 11-check linter (orphans, contradictions, stale, vocabulary drift, dataset-without-license, discovery-log, …) and suggests a fix for each issue. See Atlas → Lint.
  • To resume after a break, ask "where were we?". The agent reads the hot://<project> resource — a one-page cache of current head, summary, open questions, and live branches.
