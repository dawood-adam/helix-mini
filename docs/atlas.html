Atlas — the wiki

The LLM-maintained research memory. Markdown on disk; typed; greppable; git-friendly; cross-project.

Layout

<workspace>/atlas/
├── ATLAS.md                 the schema, in plain English (the only config agents read)
├── index.md                 one line per page
├── log.md                   chronological write log
├── inbox/                   drop zone + .manifest.json
├── raw/                     ingested originals (immutable)
├── sources/                 type: source pages
├── concepts/                type: concept pages
│   ├── contradictions/
│   ├── methods/
│   └── glossary/            workspace-shared ubiquitous language
├── entities/
│   └── datasets/            data thread anchors (E)
└── projects/<id>/
    ├── spec.md              project spec (B + F.2/F.3)
    ├── plan.md              validation plan
    ├── threads/             longitudinal artifacts (E)
    │   ├── hypothesis.md  data.md  design.md  code-org.md  glossary.md
    ├── reports/<stage>.html self-contained HTML reports (D)
    ├── syntheses/<slug>.md  via hx_atlas_save
    ├── decisions.md         the decision log
    └── _hot.md              session working state (regen at run end)

The whole tree is markdown + YAML. ATLAS.md documents the schema in plain English and is the only configuration the agents read — edit it to change conventions.

Page schema

Every page is markdown with a YAML frontmatter header:

---
id: atlas:concepts:rppg              # derived from the path
type: concept                        # concept | entity | source | method | finding | comparison
tier: canonical                      # scratch | active | canonical | published | archived
aliases: [rPPG, "remote photoplethysmography"]   # ≥ 1, mandatory
created_at:        2026-04-12T11:08:00+00:00
updated_at:        2026-05-19T15:42:00+00:00
claim_valid_at:    2026-04-12T11:08:00+00:00     # when the strongest claim was made
last_verified_at:  2026-05-19T15:42:00+00:00     # when reconciled against sources
provenance:
  stage: scout
  run_id: run_4a2c1f
  snapshot_id: "7"
  sources: [sources/lazzaro-2024.md]
links:
  derived_from: []
  related_to:   [atlas:concepts:ppg]
  contradicts:  []
  cites:        [atlas:sources:lazzaro-2024]
embeddings:
  model: BAAI/bge-small-en
  hash:  3f1c…
spec_refs: [spec:0001:G.1]            # which spec line this write supports
history:                              # short audit trail (capped at 20)
  - {at: 2026-04-12T…, action: ADD, because: "first cut from Scout", run_id: run_4a2c1f}
  - {at: 2026-05-19T…, action: UPDATE, because: "added Lazzaro 2024", run_id: run_7e91d2}
---

# rPPG

[free-form body]

Three details matter:

  • Tiers are a maturity ladder. scratch → active → canonical → published → archived. A scratch note is not a published finding; hx_atlas_promote moves a page up and confirms at canonical / published.
  • Aliases (≥ 1, enforced) drive duplicate detection across abbreviations and spellings. If an agent omits them, the title is used as the default — which the linter then flags as un-aliased.
  • The bi-temporal pair (claim_valid_at + last_verified_at) is what makes "stale" a fact rather than a guess.

Writes are backward-tolerant: an agent may emit just {path, title, content, summary} and the store fills sensible defaults — the schema never blocks a run. All writes pass the sandbox first.

Write protocol

Every Atlas write declares what it is doing, why, where it came from, and which spec line it supports. This is the Workstream-G contract (mem0 / Letta / Graphiti shape, adapted to research):

FieldRequiredWhat
pathyesAtlas-relative, must start with sources/ / concepts/ / entities/ / projects/.
titleyesPage title (control-stripped, length-capped).
contentyesMarkdown body (size-capped + size-truncated).
summaryyesOne-line index entry shown in atlas/index.md.
actionrecommendedOne of ADD · UPDATE · SUPERSEDE · LINK · NOOP. Missing → defaults to ADD with a "agent-emitted (no action declared)" rationale + a warning.
becauserecommendedA one-line rationale for the write. Read at the gate.
provenancerecommendedA small dict; the sandbox whitelists {stage, run_id, snapshot_id, confidence, sources}.
spec_refsrecommendedList of spec-line ids this write supports / contradicts / extends. Read by the orphan-by-policy lint.
type / tier / aliases / linksoptionalOverride the defaults inferred from the path. The frontmatter sanitiser normalises lists and strips control chars.

Atlas.write persists spec_refs on the page itself and appends a per-write entry to a short history array (capped at 20) so a page carries its own audit trail across iterations. The Atlas log.md records every batch with the caller-supplied message (e.g. scout | bp-from-rppg).

Pre-flight dedup with hx_atlas_propose
Before an ADD, the agent can call hx_atlas_propose(write_json). It runs the sandbox to preview rejections, runs recall on the title + aliases, and either returns "ok: no near duplicates" or a DEDUP warning with the near-matching page(s) and a suggestion to switch the action to UPDATE / SUPERSEDE / LINK. The tool does not write — it is the dedup gate the agent runs before committing.

Ingest

Drop files (PDF, markdown, text, code, data) into atlas/inbox/ and run hx_atlas_ingest — the whole inbox, or one file by path.

flowchart LR D[atlas/inbox/file.pdf] --> H{sha256 in .manifest.json?} H --yes--> SKIP[skip · already ingested] H --no--> X[extract text · pymupdf for PDFs] X --> W[Atlas.write → sources/<slug>.md] W --> M[move original → atlas/raw/] W --> U[update .manifest.json]

A .manifest.json records each file's sha256, so re-running is idempotent and a changed file is re-ingested. Each new file becomes a type: source page; the original moves to atlas/raw/, keeping the inbox a clean drop zone. Symlinks pointing outside the inbox are rejected by sandbox.validate_ingest_source.

The graph

Page links: form a three-level structure:

  • Episodes — the immutable sources in raw/.
  • Entities — the pages, with typed edges declared in frontmatter (derived_from · related_to · contradicts · cites).
  • Communities — thematic clusters, computed on demand by label propagation; not stored.

The markdown is the source of truth. atlas_index.build materialises the edges into an in-memory SQLite graph, rebuilt per query (milliseconds at this scale), so Atlas.write carries no index cost. hx_atlas_neighbors(page, k) returns the k-hop neighbourhood of a page.

Recall & get

hx_atlas_recall is one entry point with a query-shape router. It returns references only — id, title, tier, ~120-char summary, score, mode — never bodies. Bodies come from hx_atlas_get and are capped (24KB). The find/fetch split protects the model's context budget.

Query shapeModeMethod
an id or pathgraphk-hop traversal from that node
"everything / overview about X"communitythe cluster around the best hit
relational ("how does X relate to Y")graphseeded from the best lexical hit
five or more wordssemanticcosine on local embeddings
short, specificlexicalBM25

If a richer mode cannot run (no embeddings, no resolvable graph node), recall falls back to lexical — it never hard-fails. Semantic mode needs vectors; the optional fastembed extra provides a local model. Without it, semantic queries fall back to lexical. Vectors are cached in .helix/embeddings.json, keyed by body sha256.

Threads

A thread is a longitudinal artifact that spans the pipeline. Each thread lives at a known Atlas path with a tiny frontmatter and a markdown body containing one ## snap-<id> section per stage transition. The status machine is Proposed → Accepted → Superseded; the reader supports bi-temporal queries.

ThreadFirst created byUpdated byAtlas path
hypothesisScout (Frame)Scout, Scout Critic, Results Criticprojects/<id>/threads/hypothesis.md
dataScout (data sub-phase)Planner, Builder, Validatorprojects/<id>/threads/data.md + entities/datasets/<slug>.md
specScout (final sub-phase)every stage on send-backprojects/<id>/spec.md
planPlannerPlanner (revisions), Builderprojects/<id>/plan.md
designPlanner / BuilderBuilder, Results Criticprojects/<id>/threads/design.md
code-orgBuilderBuilder (refactors)projects/<id>/threads/code-org.md
glossaryScout (lit dive)every stageprojects/<id>/threads/glossary.md + concepts/glossary/<term>.md

Read the current state of a thread via the MCP resource:

thread://<project>/<name>                  # current state
thread-at://<project>/<name>/<snap_id>    # bi-temporal — state at that snapshot

Under the hood, threads.append_update(atlas_root, project, name, snapshot_id, body, status?, contributor?) is idempotent on snapshot_id: a second call with the same id replaces the section in place rather than duplicating — so a send-back re-run cleanly overwrites the previous attempt. threads.read_at truncates the body to up-to-and- including the matching ## snap-<id> section.

Glossary & ubiquitous language

Domain-Driven Design's Ubiquitous Language, realised as a thread: one term per ### <term> heading in the glossary thread, with a one-sentence definition. Workspace-shared terms live at atlas/concepts/glossary/<term>.md.

Scout seeds the glossary from the literature dive. Every later stage's prompt includes it. The lint check vocabulary-drift flags boldface terms (**term**) used in the project's spec.md / plan.md that aren't defined in the glossary — a one-click suggestion to either add the term or rename to an existing one.

Datasets as a first-class thread

In most agentic-research systems datasets are treated as fixed inputs. For computational research that is a hole. Helix makes data a first-class artifact, tied to the data thread, with per-stage instructions:

StageData-aware sub-instruction
ScoutData scout: identify candidate datasets; record each as entities/datasets/<slug>.md with {source, license, size, format, access, suitability, limitations}.
Scout CriticCritique data choice vs. the question; flag unsuitable datasets in the overlay.
PlannerData plan: which datasets, preprocessing, splits, ground-truth, validation bands tied to data.
BuilderData build: scripts that fetch / transform / version the data; partial datasets are snapshots on the data thread.
ValidatorDeterministic checks against the data spec (schema, splits respected, bands met).
Results CriticGeneralization, leakage, fairness checks tied to the data thread.

Two lint checks back this up: dataset-without-license and dataset-without-source fire when a entities/datasets/*.md page omits the field in its frontmatter — reproducibility and reuse are auditable at a glance.

Lint

hx_atlas_lint is the hygiene sweep. Each check is exact or a cheap heuristic, never fuzzy, so the output is trustworthy. The current set:

KindTriggers when
orphanA page nothing links to and that links nowhere (and isn't published / archived).
contradictionAn explicit contradicts edge between two pages.
stalelast_verified_at overdue (> 30 days); counts newer sources since.
missing-pageAn internal link pointing at a page that doesn't exist.
unaliasedOnly the auto-default alias (the title).
orphan-communityA cluster of ≥ 3 with no synthesis / comparison page.
stale-reportA non-canonical report filename under projects/<id>/reports/.
orphan-by-policyA page touched by the write protocol (has a history entry) but lists no spec_refs.
dataset-without-licenseA page under entities/datasets/ with no license field.
dataset-without-sourceSame, no source field.
vocabulary-driftA boldface term in spec.md / plan.md not defined in the project glossary.
discovery-logA hypothesis marked Supported in the hypothesis thread with no support: citations.

Each issue carries a suggested fix. hx_atlas_reconcile is the periodic sweep: it runs lint, groups findings by kind, and additionally surfaces promotion candidates (active pages with ≥ 3 inbound edges) for human review.

Hot cache

At the end of a run, atlas/projects/<id>/_hot.md is regenerated from the snapshot trail (zero-LLM — it reuses the last Decision Card): current head, summary, open questions, the directive for the next stage, recently touched stages, and live branches. It is a cache overwritten each run, distinct from the append-only snapshot history.

Read the hot://<project> MCP resource first when resuming. The agent treats it as the "where were we?" answer.
