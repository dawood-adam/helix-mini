Conceptual architecture

What Helix is, why it's shaped this way, and how to read the rest of the docs.

The shape

Helix is a thin, opinionated server that orchestrates a research project the way a good lab does:

  1. Read the room (Scout).
  2. Red-team the framing (Scout Critic).
  3. Plan a validation (Planner).
  4. Build it (Builder, test-first).
  5. Check it deterministically (Validator).
  6. Decide what it means (Results Critic).

There is a checkpoint at every transition. The pipeline pauses, shows you what just happened, and waits for you to proceed, send back with a directive, or stop. Every transition is captured as a snapshot — a tiny, content-addressed JSON record — so the history is a real version-control tree and the run is resumable from any point.

Outside the pipeline lives the Atlas: a markdown wiki, on disk, with typed YAML frontmatter. Every stage reads from it and writes to it. A second project in the same workspace starts with the first project's knowledge already present.

Three things Helix is

It helps to read Helix as three things that share one server:

Pipeline

Forge

Six stages with a structured output contract per stage and a Decision Card every stage emits — the single human-readable record. Each transition is a checkpoint where you stay in control.

Wiki

Atlas

The cross-project memory. Typed pages with maturity tiers (scratch → active → canonical → published → archived) and a closed write vocabulary (ADD · UPDATE · SUPERSEDE · LINK · NOOP) so every write is auditable.

Version control

Snapshots

A content-addressed DAG. Branch to explore alternatives. Freeze to publish. Fork to share — the whole history packs into one .tar.gz you can hand to a collaborator.

Two loops

Underneath the six stages, two loops run in concert:

  1. The execution loop — Scout, Critic, Plan, Build, Validate, Review — that any research project walks once.
  2. The discovery loop — propose → validate → revise — which iterates across the execution loop, carrying a hypothesis thread forward each iteration. Activated by autonomy_until: "rediscover"; capped at three iterations to bound spend.

The discovery loop is the same six stages, run again, with the prior iteration's outcomes (supported / refuted / inconclusive hypotheses) carried in the hypothesis thread. This is the Co-Scientist / Sakana shape, priced for a laptop.

Layered memory

Three layers of memory work together; each has a clearly defined scope:

LayerScopeWhat it holdsLifetime
Pipeline stateper runCurrent stage, sources, candidate approaches, plan, artifacts, results, verdict, feedback queue.Serialized into every snapshot.
Threadsper projectLongitudinal artifacts (hypothesis · data · spec · plan · design · code-org · glossary) updated by every stage. One YAML-frontmatter page per thread, with one ## snap-<id> section per snapshot.Survives across runs of the same project; bi-temporal (read at any snapshot).
AtlasworkspaceSources, concepts, entities, datasets, methods, glossary terms — typed and tiered. The cross-project knowledge base.Persists indefinitely; pruned via lint and the tier ladder.

The Atlas is the long-term memory; threads carry per-project evolution; pipeline state is the short-term scratchpad of one run. All three are serialized as plain markdown / YAML on disk — git- friendly, greppable, and human-readable.

Why MCP, why agent-driven

Helix is an MCP server — one stdio process the client launches on demand. That choice does three things at once:

  • Consent. The client (Claude Code) shows a confirmation before any state-changing tool runs. Helix doesn't have to reinvent a permission model.
  • Tool catalog. The agent sees a small, curated set of tools and resources — no general code-exec, no arbitrary shell. The agent has to use Helix's vocabulary.
  • Composition. Whatever skills / plugins the user's Claude Code has installed (WebSearch, WebFetch, simplify, security-review, …) are available to Helix's agents transparently — the agent uses them in its own turn.

And it's agent-driven, not sampling-driven, for a pragmatic reason: Claude Code does not implement MCP sampling. Inverting the dependency — Helix hands the prompt back, the agent answers, Helix continues — solves the problem without adding credentials, providers, or processes. The agent is the model, on the user's own plan; the server holds no keys.

Trust boundaries

Three boundaries to keep in mind:

Outer

You ↔ the agent

You talk to Claude Code in natural language. The agent decides when to call which tool. The MCP client surfaces a confirmation before any state-changing tool fires.

Middle

Agent ↔ the server

The agent calls a fixed catalog of MCP tools. Every tool that writes to disk passes its arguments through the sandbox module — names through validate_project_name, content through sanitize_atlas_writes / sanitize_code_artifacts.

Inner

Server ↔ the filesystem

The Atlas is markdown on disk; snapshots are JSON + content- addressed objects. Atlas._safe_resolve defends every Atlas path against traversal; spec_path and _resolve_path (for threads) do the same for their files.

Related work

Helix doesn't try to be a research lab in a box. It is a thin scaffold that combines patterns from agentic engineering, knowledge-graph literature, and decision-oriented programming:

  • GitHub Spec Kit — the seven-step engineering cycle Helix mirrors at the prompt level.
  • Aporia's Decision-Oriented Programming — hx_clarify is its named-question elicitation pattern.
  • STORM, Elicit, OpenAI Deep Research, Sakana AI Scientist-v2, Google AI Co-Scientist, FunSearch, Coscientist — Helix's discovery loop sits in this neighbourhood, intentionally simpler (six stages, a hard iteration cap, all running on the user's own model).
  • Graphiti — the bi-temporal pair (claim_valid_at, last_verified_at) on every page; threads' thread-at:// bi-temporal read.
  • Domain-Driven Design's Ubiquitous Language — the glossary thread + vocabulary-drift lint.
