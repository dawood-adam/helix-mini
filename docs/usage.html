Usage guide

Helix is driven through an MCP client. This guide uses Claude Code throughout.

Mental model

You talk to Claude Code in natural language. Claude Code calls Helix's MCP tools for you. When a stage needs the model, Helix returns the stage's prompt as the tool output — Claude Code is the model and answers in its own turn. The server runs no sampling and stores no credentials.

flowchart LR U[You] -- chat --> A[Claude Code] A -- hx_step / hx_submit / ... --> H[helix-mcp] H -- elicit at gates --> A A -- relay --> U H --> FS[(workspace files)] classDef u fill:#ddf4ff,stroke:#0550ae; classDef s fill:#dafbe1,stroke:#1a7f37; class A u class H s

Two things deserve emphasis:

  • The agent is the model. Helix renders each stage's SYSTEM + USER prompt and returns it as a tool result. The agent reads it, reasons, and replies via hx_submit. No sampling/createMessage, no API keys, no provider config.
  • The pipeline pauses at every gate by default. A gate is an MCP elicitation — a small structured question — that you answer through Claude Code. The Plan (Gates → Plan) controls which gates auto-proceed.

A day with Helix

A representative session, beginning to end, all in natural language:

you>   start helix
helix> [wizard] name?  question?  control mode?  source folder?
       …Scout finished. 2 candidate approaches; recommends #1.
       Proceed, send back, or stop?
you>   send back to scout — restrict to 2024 papers only
helix> Re-ran Scout with that directive. Scout Critic done. Proceed?
you>   run autonomously up to the validator
helix> …Planner done. Builder done. Validator pass. Stopping at the
       results critic per your plan. Proceed, send back, or stop?
you>   proceed
helix> Done.  verdict=ship  stages=6  tokens~12,400
you>   freeze that snapshot as v1 and fork the project as v1-public
helix> Frozen snap-7 as 'v1'. Bundle: forks/v1-public.tar.gz (124KB).
you>   health-check the atlas
helix> 3 issue(s): 2 stale, 1 missing-page. Suggestions: …
you>   resolve the missing page about Lazzaro 2024
helix> [drafts a page, asks confirm at canonical tier]
you>   confirm
helix> Promoted atlas:concepts:lazzaro-2024 → canonical.

Control modes

The control mode you pick in the wizard becomes a Plan that governs every gate:

ModeBehaviourConstructor
Step-by-stepPause and ask at every transition (default).autonomy_until=""
Auto until a stageAuto-proceed until that stage, then ask.autonomy_until="builder"
Fully autonomousNever ask; deterministic routing only.autonomy_until="END"
Rediscover loopSame as "END", plus restart from Scout after Results Critic — up to 3 iterations — carrying the hypothesis thread forward. See Gates → Rediscover.autonomy_until="rediscover"
Per-stage stepsAn explicit list of {agent, directive?, autonomy} via hx_run_plan_set(steps_json=…).—

The plan can be changed mid-run with hx_run_plan_set; the new plan is effective at the next gate. Deterministic routes (a hard Validator band miss, a Results-Critic iterate verdict) fire regardless of autonomy.

Sending a run back

Send-back is the single mechanism for course-correction. At any gate you may pick a target stage and write a one-line directive:

sequenceDiagram participant U as You participant H as helix-mcp H->>U: After scout_critic: ship · iterate · send_back? U-->>H: send_back → scout, "restrict to 2024 papers only" H->>H: record_feedback(state, from='scout_critic', to='scout', note) H->>H: snapshot the send-back H->>U: NEEDS MODEL — scout (with feedback block in USER prompt)

The directive is appended to the target stage's USER prompt as a Human feedback block on the re-run. The send-back is itself a snapshot, so the branch you took is fully resumable.

HTML reports + annotations

Every completed stage writes a self-contained HTML report at atlas/projects/<id>/reports/<stage>.html — no sidecar, no JS dependencies, the annotation JSON is embedded in the file itself. The gate prompt includes a clickable file:// link to the latest report.

Open it in Chrome / Edge. The Scout Critic's notes appear as Distill-style side margin annotations. You can:

  • Accept / Reject a critic note (the rejection reason is recorded so the agent doesn't reintroduce it).
  • Reply with a Google-Docs-style threaded comment.
  • Add a comment of your own.
  • Send back ↩ — mark the stage for re-run with this note as the directive.

Save the file (File System Access API on Chromium, with a Download-annotated-copy fallback elsewhere). Then in chat: 

you>   send back the scout report
helix> Extracted 3 annotations (1 send-back, 1 kept, 1 rejected).
       Feedback for re-run of 'scout': …

Internally this is hx_report_send_back(folder, project, stage): it parses the embedded W3C Web Annotation JSON, buckets by state, and builds the agent-readable feedback string that's threaded into the next Scout pass.

Ingesting sources

Three ways to bring material in:

  1. Direct. Put PDFs / markdown / text / code / data anywhere under the project. Scout reads them on its first turn (core.ingest.ingest_folder walks the project source folder and skips binaries).
  2. Managed via inbox. Drop files into atlas/inbox/ and ask "process my inbox". Helix runs hx_atlas_ingest: idempotent (sha256 manifest); each new file becomes a type: source page; the original moves to atlas/raw/. Re-running is a no-op.
  3. Direct write. The agent can call hx_atlas_put with explicit {path, title, content, summary, action, because, provenance, spec_refs}. The write protocol — see Atlas → Write protocol — enforces a closed action vocabulary and mandatory provenance.

Recall & get

hx_atlas_recall is one entry point with a query-shape router; hx_atlas_get is the body fetcher. Recall returns references only (id · title · tier · ~120-char summary · score · mode); bodies are capped and fetched on demand. This find/fetch split keeps a search from flooding the model's context.

You askRouter picks
"how does rPPG relate to PPG?"graph (relational query)
"overview of cuffless BP estimation"community (cluster around best hit)
"a long descriptive query of 5+ words…"semantic (cosine on local embeddings)
"short specific phrase"lexical (BM25)
atlas:concepts:rppg or a pathgraph (k-hop from that node)

Semantic mode needs vectors. The optional fastembed extra provides a local model; without it semantic queries fall back to lexical (recall never hard-fails). Vectors are cached in .helix/embeddings.json, keyed by body sha256 — only changed pages are re-embedded.

Observing & resuming

Helix is engineered so you can always answer "where were we?":

Tool / resourceTells you
hx_run_status(project)Latest run's state — status, current stage, last snapshot, token estimate, note. Persisted; survives a restart.
hx_run_events(project, since=N)The run's transition events since sequence N.
snapshot_list(project)The DAG, git-log style.
snapshot_show(project, id)One snapshot's state.
snapshot_diff(project, a, b)Tracked differences between two snapshots.
snapshot_timeline(project)Mermaid gitGraph of the history.
hot://<project>The hot cache — one-page working state regenerated at run end.
thread://<project>/<name>A project thread (hypothesis · data · spec · plan · design · code-org · glossary). Add thread-at://<project>/<name>/<snap> for a bi-temporal read.

To resume:

you>   resume bp-from-rppg from snapshot 5 on a branch called 'rppg-v2'

# under the hood:
resume_pipeline(project="bp-from-rppg", snapshot=5,
                branch="rppg-v2", folder="./bp-from-rppg")

resume_pipeline rebuilds the snapshot's full state (artifact bytes come from the content-addressed object store), re-enters the agent-driven loop at the snapshot's stage by default (or any other stage via at=), and returns the next prompt. folder self-roots the run at the project, regardless of the server's cwd.

Budget & pauses

Per-project helix.toml: 

[limits]
token_cap = 200000     # ≈ chars/4 estimate, across all stages
call_cap  = 50         # number of LLM-stage submissions

Cycling has no fixed cap. Reaching either ceiling pauses the run with a resumable snapshot; interactively, you are offered the choice to raise the ceiling and continue. The run does not crash and does not lose progress — every stop has a snapshot.

Other pause causes (all resumable):

  • Declined gate elicitation (Declined).
  • Lost MCP client mid-stage (ClientUnavailable) — snapshot the partial work and stop.
  • Stage error — snapshot with decision: "error" and return.
  • Server restart between hx_step and hx_submit — the pending step is persisted to .helix/runs/<project>.pending.json; the next hx_step picks it back up.

Publishing a result

The "freeze and fork" checklist when a run is ready to share:

  1. hx_atlas_lint — resolve blockers.
  2. hx_atlas_promote(ids, "published") — confirms via elicitation.
  3. hx_snap_freeze(project, snapshot, tag) — mark the snapshot immutable.
  4. hx_snap_fork(project, name) — exports forks/<name>.tar.gz (snapshots + objects + index + refs). One file; reproducible.

Or invoke the canonical workflow by name: helix_freeze walks you through it.
